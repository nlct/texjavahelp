\begin{prefrontmatterblock}{note}{Note}
The \gls{TeXJavaHelp} System is provided \qt{as is}. It's limited to a
few supported packages. It's designed to integrate into a Java
application's in-application manual and context-sensitive help
buttons (as well as providing a PDF for general use). 
Since Java's HTML and CSS support is quite limited, this
means that the files generated by the \gls{TeXJavaHelp} System must be
able to comply with this limitation.

\begin{warning}
This manual is incomplete and the library is still a work in
progress. The library is provided as I have Java applications that
require it and therefore need to have it installed.
\end{warning}

\end{prefrontmatterblock}

\chapter{Introduction}
\label{sec:intro}

I have a number of Java \gls{gui} applications, which require a manual.
Usually I create \gls{pdf} manuals using \pdfLaTeX\ or \LuaLaTeX, but
it's useful for a \gls{gui} to also have an in-application help, where 
an appropriate section can be opened for context-sensitive help (for example,
via a \widgetfmt{Help} button in a dialog).

In the past, I used Oracle's \gls{JavaHelp}\texttrademark\ but it has a number of
limitations and now no longer seems to be supported. Ideally, I
would like the same document source to provide the \gls{pdf} (for
access outside of the application), and the \gls{html} and \gls{xml}
files required for the application's help system, and (if the
documentation is particularly large) \gls{html} files for the
application's home page.

I've tried several ways to achieve this. The first was to use \LaTeX\ source
and convert to \gls{html} using a general purpose \LaTeX\ to
\gls{html} converter. The limitations of Java's HTML and CSS
support, the complexity of adding support for \gls{JavaHelp}['s] custom
\xmltagfmt{object} format, and the need to create the supplementary \gls{xml}
files, meant that some additional work was required to get the correct output.

The second method I tried was to start with \gls{xml} source with a custom
script which generated both the \gls{html} and \gls{xml} files for \gls{JavaHelp}
and also the \LaTeX\ source for the \gls{pdf}. This proved the more reliable route
for the \gls{html} and \gls{xml} output, as it's much easier to convert from \gls{xml}
to \gls{html}, but I don't particularly like writing large documents in 
\gls{xml}.

One of the advantages of \gls{html} over \gls{pdf} is that it's more
accessible.  This should mean that the in-application help ought to
be more accessible than the \gls{pdf} manual, but I couldn't find a
way with \gls{JavaHelp} to allow the user to customise the font
without having to resort to editing the \gls{html} or \gls{css} files (for
example, to make the font larger for visually impaired users or to
change to a dyslexic font).

I'm now trying a different approach. Since most of my Java applications are in
some way related to \TeX, and I created the \gls{TeXParserLib} to parse
\LaTeX\ code to assist them, I decided to try using the \gls{TeXParserLib}
to parse the \LaTeX\ source code for the manual in order to create the 
\gls{html} and \gls{xml} files needed for a \gls{gui} help system. At the same
time, I decided to also create a library to provide a simple in-application help
system to replace \gls{JavaHelp}.

The biggest drawback is that the \gls{TeXParserLib} was originally designed to
perform limited parsing of documents: for extracting information, removing obsolete
or problematic code, or for converting short fragments into \gls{html}.
It wasn't intended for converting entire documents to \gls{html} and lacks support
for all but a handful of packages. The available support is essentially limited
to the support required for the various projects I've worked on. It's now 
principally geared towards \app{bib2gls} and \app{datatooltk}.

However, I have started using the \gls{TeXParserLib} to convert my \LaTeX\
package user manuals to \gls{html} to test the parsing capabilities. This means
that there's already support for my \sty{nlctuserguide} package. So I've started
by creating a new package, \styfile{texjavahelp}, that's heavily based on
\styfile{nlctuserguide} so that I can reuse much of the \gls{TeXParserLib}
support for \sty{nlctuserguide}. Unlike \sty{nlctuserguide}, which is 
designed for a self-contained source document, the \sty{texjavahelp} package
is designed to have accompanying \ext{bib} files for use with \app{bib2gls}.

The \glsxtrfull{TeXJavaHelp} repository is at \url{https://github.com/nlct/texjavahelp}
and consists of:

\begin{itemize}
\item \file{texjavahelplib.jar}: the
\gls{TeXJavaHelpLib}, which provides a \gls{gui} 
component to display the \gls{html} files created by \app{texjavahelpmk}
and localisation support for messages and widgets (menus, buttons,
etc).

\item \styfile{texjavahelp}: a \LaTeX\ package to help create the \gls+{pdf} version of
the documentation from the \LaTeX\ source. Requires the \sty{glossaries-extra}
package and \app{bib2gls}. This needs to be on \TeX's path.

\item \file{texjavahelplibicons-16.def} and
\file{texjavahelplibicons-16.pdf} size 16 icon image mapping and PDF
files. These need to be on \TeX's path.

\item \file{texjavahelplibicons-20.def} and
\file{texjavahelplibicons-20.pdf} size 20 icon image mapping and PDF
files. These need to be on \TeX's path.

\item \file{texjavahelplibicons-24.def} and
\file{texjavahelplibicons-24.pdf} size 24 icon image mapping and PDF
files. These need to be on \TeX's path.

\item \file{texjavahelplibicons-32.def} and
\file{texjavahelplibicons-32.pdf} size 32 icon image mapping and PDF
files. These need to be on \TeX's path.

\item \file{texjavahelplibicons-64.def} and
\file{texjavahelplibicons-64.pdf} size 64 icon image mapping and PDF
files. These need to be on \TeX's path.

\item \app{tjhxml2bib}: a \gls{cli} application that converts
localisation resource files to \ext{bib} files suitable for use with
\app{bib2gls}.

\item \app{tjhcreateiconpdf}: a \gls{cli} application that creates a
PDF file containing each icon image on a separate page and a mapping file.
This allows both the \LaTeX\ source code and the helpset interface
to access and display icon images that are bundled in a \ext{jar} file.

\item \app{texjavahelpmk}: a \gls{cli} application that uses the 
\gls{TeXParserLib} (with a custom implementation of the \sty{texjavahelp} package) 
to create the \gls+{html}, \gls+{css} and \gls+{xml} files (required by the 
\gls{TeXJavaHelpLib}) from the \LaTeX\ source.

\item \app{tjhflattendocsrc}: a \gls{cli} application that uses the
\gls{TeXParserLib} to create a flattened version of the \LaTeX\ document
source for compact standalone distribution (that is, any instance of
\gls{input} is replaced with the file content);

\item \app{tjhziphelpset}: a \gls{cli} application used to bundle
the helpset files (and standalone \gls{html} license file, if applicable) into
a \ext+{tjh} file (which is a special type of \ext+{zip} file).
\end{itemize}

The \app{texjavahelpmk} application is much more limited than
general purpose \LaTeX\ to \gls{html} applications, but it does more
than simply convert from \LaTeX\ to \gls{html}. The \ext{xml} files
that it also creates are an integral part of the \gls{gui} system
provided by \file{texjavahelplib.jar}. The indexing data
(\file{index.xml}) is obtained via the \ext{glstex} files created by \app{bib2gls} 
and the \ext{aux} file, the navigation data (\file{navigation.xml}) is obtained 
by parsing the \ext{aux} file, and the search data (\file{search.xml})
is created while \app{texjavahelpmk} processes each paragraph.

This means that \app{texjavahelpmk} is likely to take longer than a
regular \LaTeX\ to \gls{html} generator, as it has to not only
convert the source \LaTeX\ code to \gls{html} code but then has to
parse and markup blocks of the \gls{html} code for the search data
and search context before writing the \gls{html} code to the output file.

\begin{information}
You will need \file{texjavaparserlib.jar} installed if you want to
use any of the \gls{cli} applications with a corresponding \TeX Lua
script. An application that just uses the helpset and localisation
support provided by \file{texjavahelplib.jar} doesn't need
\file{texjavaparserlib.jar} to run (unless it requires any \TeX\
parsing independently of the \gls{TeXJavaHelp} System).
\end{information}

\chapter{Example}\label{sec:example}

For a complete but small example application, see \appfmt{jmakepdfx}
(\url{https://github.com/nlct/jmakepdfx}) which can either be run in
\gls{gui} mode or in \gls{cli} batch mode.
The build steps for \appfmt{jmakepdfx} are:
\begin{enumerate}
\item Compile the Java code with \file{texjavahelplib.jar} on the
Java class path.
\item Create a \ext+{jar} file that includes the dictionaries
(replace \code{path/to} as appropriate):
\begin{terminal}
jar cmf path/to/Manifest.txt path/to/jmakepdfx.jar \bkslnl
com/dickimawbooks/jmakepdfx/*.class \bkslnl
com/dickimawbooks/jmakepdfx/dictionaries \bkslnl
com/dickimawbooks/jmakepdfx/dictionaries/*.xml \bkslnl
com/dickimawbooks/jmakepdfx/icons/*.png
\end{terminal}
\item Create the \app{bib2gls} file from the applicable localisation
files (replace \code{path/to} as appropriate):
\begin{terminal}
cd doc
\app{tjhxml2bib} \bkslnl
\TJHXBswitch{resource} \bkslnl
/com/dickimawbooks/texjavahelplib/dictionaries/texjavahelplib-en.xml \bkslnl
path/to/dictionaries/jmakepdfx-en.xml \bkslnl
\TJHXBswitch{output} jmakepdfx-dict-en.bib
\end{terminal}
\item Build the PDF file (\LuaLaTeX, \app{bib2gls}, \LuaLaTeX, \LuaLaTeX).
\item Create the helpset files (change \code{path/to} as applicable):
\begin{terminal}
\app{texjavahelpmk} jmakepdfx.tex path/to/helpset
\end{terminal}
\item Bundle the helpset files and the license file (where
\code{path/to} is as in the previous step):
\begin{terminal}
\app{tjhziphelpset} \bkslnl
\TJHZHswitch{license-file} gpl-3.0-standalone.html en \bkslnl
\TJHZHswitch{in-dir} path/to \bkslnl
\TJHZHswitch{output} path/to/jmakepdfx-helpset.tjh
\end{terminal}

\end{enumerate}

 \gls{label}\marg{listoftables}\comment{label also HTML basename}
}

The label following a section heading is used as the basis for the
\gls{html} base name. A colon prefix, such as \qtt{sec:} will be
stripped and other awkward characters will be replaced.

The abstract class \code{AbstractCLI} is provided for applications
that can be started from a command prompt or terminal with switches
(such as \longargfmt{in} or \shortargfmt{i}). This abstract class
sets up the \code{CLISyntaxParser} with common switches
such as \longargfmt{help} (print help and exit), 
\longargfmt{version} (print version and exit),
\longargfmt{verbose} (enable verbose mode), 
\longargfmt{noverbose} (disable verbose mode) and 
\longargfmt{debug} (debugging information).

There are methods provided for these switches:
\begin{codebox}
public void printCLIAbout()
public String getCLIApplicationName()
public String getCLIApplicationVersion()
public String getCLIApplicationVersionDate()
public void printCLISyntax()
\end{codebox}

For additional switches, you will need to specify the number of
arguments that should follow the switch. This is done by overriding:
\begin{codebox}
protected int getCLIArgCount(String arg)
\end{codebox}
The \code{arg} argument will be the switch (for example, 
\longargfmt{in} or \shortargfmt{i}). The return value should be the
number of arguments that must follow the switch. For example:
\begin{codebox}
protected int getCLIArgCount(String arg)
\marg{
    if (arg.equals("--in") || arg.equals("-i")
     || arg.equals("--output") || arg.equals("-o")
      )
   \marg{
      return 1;
   }
\codepar
   return 0;
}
\end{codebox}
This indicates that \longargfmt{in}, \shortargfmt{i},
\longargfmt{output}, \shortargfmt{o} all require one argument and
all other switches don't have an argument.

If you have a switch that requires more than one argument, you will
need to increase the maximum allowed number by overriding:
\begin{codebox}
protected int getCLIMaxArgParams()
\end{codebox}
The \app{tjhziphelpset} application needs to do this to allow for
the \TJHZHswitch{license-file} switch which takes two arguments:
\begin{codebox}
@Override
protected int getCLIMaxArgParams()
\marg{
   return 2;
}
\end{codebox}

Each switch is processed using:
\begin{codebox}
protected boolean parseCLIArg(String arg, CLIArgValue[] returnVals)
  throws InvalidSyntaxException
\end{codebox}
This should return true if the switch given by \code{arg} is valid
and false otherwise. The \code{returnVals} argument contains the
switch parameters.

Valueless switches can simply use \code{arg.equals("\ldots")}
but switches with values need to use one of the one of the provided 
\code{isArg}, \code{isIntArg}, \code{isListArg} etc methods.
For example:
\begin{codebox}
@Override
protected boolean parseCLIArg(String arg, CLIArgValue[] returnVals)
throws InvalidSyntaxException
\marg{
   if (arg.equals("--gui") || arg.equals("-g"))
   \marg{
      guiMode = true;
   }
   else if (arg.equals("--nogui"))
   \marg{
      guiMode = false;
   }
   else if (isArg(arg, "--in", "-i", returnVals))
   \marg{
      if (returnVals[0] == null)
      \marg{
         throw new InvalidSyntaxException(
            getMessage("error.clisyntax.missing\_value", arg));
      }
\codepar
      inFile = new File(returnVals[0].toString());
   }
// ...
   else
   \marg{
      return super.parseCLIArg(arg, returnVals);
   }
\codepar
   return true;
}
\end{codebox}
An argument that doesn't follow a single or double-hyphen switch
is processed with:
\begin{codebox}
@Override
protected void parseNoSwitchCLIArg(String arg)
throws InvalidSyntaxException
\end{codebox}
For example:
\begin{codebox}
@Override
protected void parseNoSwitchCLIArg(String arg)
throws InvalidSyntaxException
\marg{  
   if (inFile == null)
   \marg{
      inFile = new File(arg);
   }
   else if (outFile == null)
   \marg{
      outFile = new File(arg);
   }
   else
   \marg{
      throw new InvalidSyntaxException(
            getMessage("error.syntax.only\_one\_inout"));
   }
}
\end{codebox}

Although the \code{AbstractCLI} class was defined for \gls{cli}
applications, it can also be used for \gls{gui} applications. Just
make sure to override:
\begin{codebox}
public boolean isGUIMode()
\end{codebox}
This should return true for \gls{gui} applications and false
otherwise. For example:
\begin{codebox}
@Override
public boolean isGUIMode()
\marg{
   return guiMode;
}
\end{codebox}

The \code{AbstractCLI} class automatically defines an instance
of \code{TeXJavaHelpLib} and an associated \code{TeXJavaHelpLibApp}.
The \file{texjavahelplib.jar} dictionaries are automatically loaded, 
so common messages provided by the \gls{TeXJavaHelpLib} are
available. For additional messages, your application will need to
provide its own set of localisation files. These files can add extra
messages or override the \gls{TeXJavaHelpLib} messages.

The files should be named \filemetametafmt{}{prefix}{-}{tag}{.xml}
where \meta{tag} is a localisation tag (such as \qtt{en} or
\qtt{en-GB}). A more specific instance will be loaded after the less
specific instance, allowing regional variations to override the
root language message.

For example, a button's text, mnemonic and tooltip could be
specified in \filefmt{example-en.xml} as:
\begin{codebox}
\xmlStartElem{entry}{\xmlattr{key}=\glsquote{button.choosecolor}}Choose Color...\xmlEndElem{entry}
\xmlStartElem{entry}{\xmlattr{key}=\glsquote{button.choosecolor.mnemonic}}C\xmlEndElem{entry}
\xmlStartElem{entry}{\xmlattr{key}=\glsquote{button.choosecolor.tooltip}}Open selector\xmlEndElem{entry}
\end{codebox}
The file \filefmt{example-en-GB.xml} just needs to replace the
button text entry:
\begin{codebox}
\xmlStartElem{entry}{\xmlattr{key}=\glsquote{button.choosecolor}}Choose Colour...\xmlEndElem{entry}
\end{codebox}

The \code{AbstractCLI} class provides a method to load additional
dictionaries. The \appfmt{jmakepdfx} application overrides the
method as follows:
\begin{codebox}
@Override
protected void loadDictionaries(MessageSystem msgSys)
throws IOException
\marg{  
   msgSys.loadDictionary(
    \glsquote{/com/dickimawbooks/jmakepdfx/dictionaries/},
    \glsquote{jmakepdfx});
} 
\end{codebox}
The dictionary files \filemetafmt{jmakepdfx-}{tag}{.xml} are embedded
in the \filefmt{jmakepdfx.jar} file.

\begin{important}
The \ext+{jar} file format is a special type of \ext+{zip} file.
The file format allows entries to be specified with a sub-path
without including the actually directory name. In this case,
the \filefmt{dictionaries} directory must be explicitly included in
the \ext{jar} file otherwise the message system can't traverse it.
\end{important}

If there is no corresponding dictionary for the specified locale
then the fallback language will be used. This defaults to English.
The fallback dictionary should always be provided.

Message text can be accessed with one of the following
\code{AbstractCLI} methods:
\begin{codebox}
public String getMessageWithFallback(String label,
   String fallbackFormat, Object... params)
public String getMessageIfExists(String label, Object... params)
public String getMessage(String label, Object... params)
\end{codebox}
The label is the value of the \xmlattr{key} attribute of the
\xmlelem{entry} element provided in the language file.
The element value should be formatted according to
\code{java.text.MessageFormat} syntax. Message arguments are
identified by \code{\marg{0}} (first argument), 
\code{\marg{1}} (second argument), etc.

The \code{AbstractCLI} class provides the method to access the help
library:
\begin{codebox}
public TeXJavaHelpLib getHelpLib()
\end{codebox}
The returned \code{TeXJavaHelpLib} variable is created within:
\begin{codebox}
public void initialiseHelpAndParse(String[] args)
throws IOException,InvalidSyntaxException
\end{codebox}
where the \code{args} array is as passed to the \code{main} method.
This method should be called as soon as possible to ensure that the
\code{TeXJavaHelpLib} variable has been initialised before it's needed. 

There are two locale settings: for the message system (which
determines the dictionary files to use) and for the helpset
(the files within the \ext+{tjh} file). The default for both
settings is the Java runtime environment's default locale.
If there is no support for the locale, the fallback will be used
instead.

If the message system locale needs to be changed, the preferred
locale can be set with one of the following \code{AbstractCLI} methods:
\begin{codebox}
public void setMessageLocale(Locale locale)
public void setMessageLocale(HelpSetLocale locale)
\end{codebox}
However, this must be done before the messaging system is initialised.
These methods will throw an \code{IllegalArgumentException} if
they are used too late.

If the helpset locale needs to be changed, the preferred locale can
be set with one of the following \code{AbstractCLI} methods:
\begin{codebox}
public void setHelpSetLocale(Locale locale)
public void setHelpSetLocale(HelpSetLocale locale)
\end{codebox}
These methods may be used after the messaging system has been
initialised but they must be used before the helpset is initialised
with the \code{TeXJavaHelpLib} method \code{initHelpSet()}. If these
methods are used too late, they will have no effect.

If the application's helpset has been bundled in the file called
\filefmt{example-helpset.tjh} then you need to identify this before
creating the help windows:
\begin{codebox}
TeXJavaHelpLib helpLib = getHelpLib();
helpLib.setHelpSetZipName("example-helpset.tjh");
helpLib.initHelpSet();
\end{codebox}
Assuming that a \code{JFrame} called \code{jframe} has been defined,
a help menu can now be added:
\begin{codebox}
JMenuBar mBar = new JMenuBar();
jframe.setJMenuBar(mBar);
JToolBar toolbar = new JToolBar();
// ...
JMenu helpM = helpLib.createJMenu("menu.help");
mBar.add(helpM);
\codepar
TJHAbstractAction manualAction = helpLib.createHelpManualAction();
helpM.add(new JMenuItem(manualAction));
toolbar.add(manualAction);
\end{codebox}
If a license \gls{html} file has been bundled in the \ext{tjh}
file (using \app{tjhziphelpset}['s] \TJHZHswitch{license-file} switch)
then a license dialog can easily be created:
\begin{codebox}
MessageDialog licenseDialog = helpLib.createLicenseDialog(jframe,
  helpLib.getMessage("license.title"));
\end{codebox}

A dialog can have a help button that opens a specific section of the
manual.  This can be created using the \code{TeXJavaHelpLib} method:
\begin{codebox}
public JButton createHelpDialogButton(JDialog owner, String helpId)
 throws IllegalArgumentException,
        HelpSetNotInitialisedException
\end{codebox}
For example, suppose a settings dialog is described in the
manual:
\begin{codebox}
\gls{section}\marg{Settings Dialog}
\gls{label}\marg{sec:settings}
\end{codebox}
then the okay, cancel and help buttons can be added:
\begin{codebox}
TeXJavaHelpLib helpLib = ...
JDialog jdialog = ...
JPanel buttonPanel = new JPanel();
jdialog.getContentPane().add(buttonPanel, "South");
\codepar
OkayAction okayAction = ...
buttonPanel.add(helpLib.createOkayButton(okayAction, jdialog.getRootPane()));
buttonPanel.add(helpLib.createCancelButton(jdialog));
buttonPanel.add(helpLib.createHelpDialogButton(jdialog, "sec:settings"));
\end{codebox}

\chapter{The \stytext{texjavahelp} \LaTeX\ Package}
\label{sec:texjavahelpsty}

\pkgdef{texjavahelp}

The \sty{texjavahelp} package requires the following packages:
\sty{glossaries-extra}, \sty{pifont}, \sty{fontawesome}, \sty{twemojis},
\sty{upquote}, \sty{hologo}, \sty{varioref}, \sty{xcolor},
\sty{graphicx}, \sty{tikz}, \sty{tcolorbox} and \sty{hyperref}.
These are loaded automatically.

The \sty{glossaries-extra} package is loaded with the options:
\optval{record}{nameref} (which means that \app{bib2gls} should be used),
\opt{indexcounter}, \opt{floats}, \opt{nostyles},
\optvalm{stylemods}{tree\dcomma book\-index\dcomma topic} and
\optval{style}{alttree}. (The \gls{TeXParserLib} only has limited
support for glossary styles.)

\begin{information}
Since \sty{texjavahelp} was adapted from \sty{nlctuserguide}, it
copies the package options and some command names from that package.

The special prefixes \qtt{menu.}\ and \qtt{button.}\ determine the
entry type (\bibentry{menu} and \bibentry{button}). The special
suffixes \qtt{.mnemonic}, \qtt{.keystroke}, \qtt{.tooltip},
\qtt{.iconimage} and \qtt{.defaultkeys} identify entry fields (\bibentryfield{mnemonic},
\bibentryfield{keystroke}, \bibentryfield{tooltip},
\bibentryfield{iconimage} and \bibentryfield{defaultkeys}).

The \bibentryfield{defaultkeys} field is a special field that
supplies default keys to apply when using commands like \gls{gls} or
\gls{glstext}. For example, the \qtt{button.help} entry will default
to using an ignored location, which means it will be selected but 
it won't have a location list unless the format is specifically
change for a particular reference. This is useful for common entries
that are referenced so often it would otherwise create a lengthy
location list, which isn't helpful.

The \sty{texjavahelp} package uses \gls{GlsXtrSetPlusModifier}
to change the behaviour of the \code{+} modifier so that the format
is set to the general \glscsname{glsnumberformat} format.
This means that \code{\gls{gls}\marg{button.help}} won't add a
location to the location list but
\code{\gls{gls}+\marg{button.help}} will.

The \sty{texjavahelp} package uses \gls{GlsXtrSetAltModifier} to set
\code{!} as an alternative modifier to set the format to
\glscsname{glsignore}. So \code{\gls{gls}!\marg{menu.help}} will
reference the \qtt{menu.help} entry (which doesn't have
\bibentryfield{defaultkeys} set) without adding a location to the
location list.

There are some convenient commands provided to reference entries and
their fields without having to include the label prefix.
These are listed in \sectionref{sec:tjhglscmds}.

\section{Semantic Commands}
\label{sec:semanticcmds}

Semantic commands are commands that relate to a particular idea or meaning.
For example, \code{\gls{emph}\margm{text}} is a semantic command that indicates
emphasis. This may render \meta{text} in an italic or slanted font, but this isn't always the
case. In \gls{html}, this is equivalent to using
\code{\xmlelem{em}\meta{text}\xmlEndElem{em}}.
The style may be changed, but the meaning remains.

Both semantic and syntactic commands have a defined syntax, but
there is no meaning associated with purely syntactic commands, such
as \gls{textit}. For example, the commands \gls{emph} and
\gls{textit} have the same syntax, in that they both take a single
argument, but \gls{emph} specifically means that the argument should
be emphasized, whereas \gls{textit} is a font changing instruction.

Font declarations should be avoided with \app{texjavahelpmk}, except where \gls{bfseries} is
used at the start of a \env{tabular} cell, which will be interpreted
as a header cell.

The \sty{texjavahelp} package provides a number of semantic commands, not only
because it's good practice but also because it helps the \gls{TeXParserLib} to
produce more accessible \gls{html} with as little inline styling as possible.
The other reason is to allow the use of label prefixes to help disambiguate
closely related labels.  For example, the label \qtt{switch.help} is for the
command line switch \switch{help} whereas \qtt{menu.help} is for a \menu{help}
menu and \qtt{button.help} is for a button (\widget{help} or
\btn{help}).

Commands that specifically relate to entries that have been defined
in a \ext{bib} file are described in \sectionref{sec:tjhglscmds}.
This section describes general semantic commands, some of which may
be used in entry fields, such as \bibentryfield{name} or
\bibentryfield{keystroke}.

\cmddef{strong}
In the event that \gls{strong} hasn't been defined, the
\sty{texjavahelp} package will provide it.
\begin{coderesult}
\gls{emph}\marg{emphasized},
\gls{strong}\marg{strong}.
\tcblower
\emph{emphasized},
\strong{strong}.
\end{coderesult}

The \app{texjavahelpmk} application on parsing this will create the HTML code:
\begin{compactcodebox*}
\xmlelem{em}emphasized\xmlEndElem{em},
\xmlelem{strong}strong\xmlEndElem{strong}.
\end{compactcodebox*}

A similar command is:
\cmddef{code}
This has its argument rendered in a monospaced font (but not
verbatim) in the PDF but \app{texjavahelpmk} will use the
\xmlelem{code} element. You can use \gls{verb} or a similar command for
verbatim inline content but be aware of its limitations.
\begin{coderesult}
\gls{code}\marg{\glssymbol{atchar} \glssymbol{cs.hash} \glssymbol{cs.percent} \glssymbol{cs.underscore}}
\codepar
\gls{verb}|@ \glssymbol{hashchar} \glssymbol{percentchar} \glssymbol{underscorechar}|
\tcblower
\code{@ \# \% \_}

\verb|@ # % _|
\end{coderesult}
The difference in the HTML file is that the \gls{code} content will
be marked up with \xmlelem{code} but the \gls{verb} content won't
have any markup.

The other advantage with \gls{code} is that you can reference
entries in it. For example:
\begin{coderesult}
\gls{code}\marg{\gls{glssymbol}\marg{atchar}
\gls{glssymbol}\marg{cs.hash}
\gls{glssymbol}\marg{cs.percent}
\gls{glssymbol}\marg{cs.underscore}}
\tcblower
\code{\glssymbol{atchar}
\glssymbol{cs.hash}
\glssymbol{cs.percent}
\glssymbol{cs.underscore}}
\end{coderesult}
The records are ignored by default for references within \gls{code}
which causes \app{bib2gls} to ensure the entries are selected but
won't add the location to the entry's location list. This helps to
avoid overly long location lists.

Other semantic commands will usually be written with a
\xmlelem{span} tag, but as Java's HTML and CSS support is quite
limited, the default \TJHMKswitch{helpset} setting will sometimes use
\xmlelem{font} to get around some of those limitations.
The \TJHMKswitch{html} setting switches on \app{texjavahelpmk}['s] HTML5
mode so in that case \xmlelem{font} won't be used.

There are some convenient shortcut commands but note that these rely
on the label being in the form \meta{prefix}\meta{tag} where
\meta{tag} is the element name. If the
\meta{prefix} optional argument is omitted, it will have the default
value \qtt{xmlelement.}\ and the starred version will insert
\code{format\dequals glsignore,} to the start of the \meta{options} list.
The \meta{attr-list} argument is a list of the element's attributes.

\cmddef{xmlStartElem}
This essentially does:
\begin{compactcodebox}
\gls{xmltagfmt}\marg{\gls{glslink}\oargm{options}\marg{\meta{prefix}\meta{tag}}\margm{tag} \meta{attr-list}}
\end{compactcodebox}
Note that this puts the \meta{attr-list} outside of the link text,
which means that it may contain attribute references without causing
interference from nested links. However, \meta{attr-list} is placed
inside \gls{xmltagfmt}.
\begin{coderesult}
\gls{xmlStartElem}\marg{img}\marg{src=\gls{glsquote}\marg{image.png}}
\tcblower
\xmlStartElem{img}{src=\glsquote{image.png}}
\end{coderesult}

\cmddef{xmlVoidElem}
In this case, the final \meta{attr-list} argument is optional. If
present, this command does:
\begin{compactcodebox}
\gls{xmltagfmt}\marg{\gls{glslink}\oargm{options}\marg{\meta{prefix}\meta{tag}}\margm{tag} \meta{attr-list}/}
\end{compactcodebox}

If the final \meta{attr-list} is omitted, this command does:
\begin{compactcodebox}
\gls{xmltagfmt}\marg{\gls{glslink}\oargm{options}\marg{\meta{prefix}\meta{tag}}\margm{tag}/}
\end{compactcodebox}
The difference between omitting the \meta{attr-list} optional
argument and setting it to an empty value is the spacing before
\meta{attr-list}.
As with \gls{xmlStartElem}, the \meta{attr-list} (if present) is outside of the
link text (but inside \gls{xmltagfmt}).
\begin{coderesult}
\gls{xmlVoidElem}\marg{hr}
\codepar
\gls{xmlVoidElem}\marg{img}\oarg{src=\gls{glsquote}\marg{image.png}}
\codepar
\gls{xmlVoidElem}\marg{hr}\oarg{}
\tcblower

\xmlVoidElem{hr}[]
\end{coderesult}

\subsection{Custom Semantic Commands}
\label{sec:customsemanticcmds}

If the provided commands are insufficient, you can define your own
custom semantic command with:
\cmddef{defsemanticcmd}
where \meta{colour} (if present) indicates a colour specification
that can be used in the argument of \gls{color}, \meta{cs} is the
command that needs to be defined (which will take a single argument), 
\meta{font-cs} (which may be empty if not applicable) should be a 
text-block font command to apply to the argument of \meta{cs}, and
\meta{prefix} (which may be empty) is a prefix to insert in front of
the argument of \meta{cs}.

Some predefined colours are provided: 
\optfmt{cs} (dark green), 
\optfmt{csopt} (DarkOrchid4), 
\optfmt{styopt} (DarkOrchid4), 
\optfmt{comment} (grey), 
\optfmt{faded} (grey), 
\optfmt{banned} (red),
\optfmt{style1} (DarkSlateGray4),
\optfmt{style2} (SteelBlue4),
\optfmt{style3} (Blue4),
\optfmt{style4} (Purple4),
\optfmt{style5} (SlateBlue4),
\optfmt{style6} (Tan4).
These are all recognised by \app{texjavahelpmk} so they can be
referenced, but you will need to change the CSS file to modify the
associated classes if you want the style changed.

For example:
\begin{codebox*}
\gls{defsemanticcmd}\oarg{purple}\marg{\csfmt{hashtag}}\marg{\gls{texttt}}\marg{\glssymbol{cs.hash}}
\end{codebox*}
This defines the command \csfmt{hashtag} with a single argument
where \code{\csfmt{hashtag}\margm{text}} will be typeset in the PDF
using:
\begin{compactcodebox}
\gls{fmtorcode}\gls{texttt}\marg{\gls{color}\marg{purple}\glssymbol{cs.hash}\meta{text}}
\end{compactcodebox}
However, \app{texjavahelpmk} will behave differently. It will create
a CSS class associated with this command and \csfmt{hashtag} will
create an element that references that class.

Note that if the \meta{font-cs} argument is not empty, the
\meta{font-cs} command will be preceded by:
\cmddef{fmtorcode}
This normally just expands to \code{\meta{cs}\margm{text}} but in a
code context, such as with the argument of \gls{code} or in the body
of \env{codebox}, this just expands to \meta{text}. That is, the
formatting command \meta{cs} is stripped and the braces are removed
from \code{\margm{text}}.

\subsection{Other Commands}
\label{sec:othercmds}

\cmddef{longswitch}
Used as a prefix for \gls{longargfmt} this expands to two hyphens
separated by a thin space to visually distinguish them (and prevents
them from becoming an en-dash).
The thin space is not included in \app{texjavahelpmk}['s] definition
of this command, which makes it easier to copy and paste.

\cmddef{shortswitch}
Used as a prefix for \gls{shortargfmt}, this expands to a single hyphen.

Long filenames, options and variables that include hyphens in their name are
awkward to hyphenate if they can't fit on a line. Preventing a line
break can cause large amounts of white space but hyphenating can
cause confusion between an actual \gls{hyphen-minus} that
forms part of the name and a hyphen introduced by breaking the name. If a suitable font is
provided to allow \gls{texttt} (used by \gls{filefmt}) to hyphenate
its argument, then you can use \gls{dhyphen} to clearly mark
actual hyphens that form part of the name.

The following commands are shortcuts for use in the \bibentryfield{syntax}
field.
\cmddef{metaboolean}
Expands to \code{\gls{meta}\marg{boolean}}.

\cmddef{keyval}
Expands to
\code{\gls{meta}\marg{key}\gls{dequals}\gls{meta}\marg{value}}.

\cmddef{keyvallist}
Expands to \code{\gls{meta}\marg{key\gls{dequals} value list}}.

The following commands are designed for filenames, command names or
options with punctuation but no spaces that may occur inline but
cause problems for paragraph justification.

\cmddef{dhyphen}
A \gls{hyphen-minus} character that allows a line break before it without placing a
hyphen before the break.

\cmddef{dequals}
An \gls{equal-sign} that allows a line break before or after it without a
hyphen.

\cmddef{dcomma}
A \gls{comma} that allows a line break after it.

\cmddef{dcolon}
A \gls{colon} that allows a line break after it without a hyphen.

\cmddef{dfullstop}
A \gls{fullstop} that allows a line break after it without a hyphen.

\cmddef{dunderscore}
An underscore (produced with \gls{cs.underscore}) that allows a line break before
or after it without a hyphen.

\cmddef{tjhsetrmhyphen}
The \gls{tjhsetrmhyphen} command is within the semantic commands
described above, such as \gls{csfmt} and \gls{filefmt}, to locally
redefine the discretionary hyphen command \inlineglsdef{cs.hyphen} to
use the Roman family hyphen so that it's more obviously not part of
content formatted in a monospaced font, since a Roman \gls{hyphen}
is typically shorter than a monospaced \gls{hyphen-minus}.
(See the \hyperlink{longfilenameexample}{earlier long filename example}.)

The \gls{tjhsetrmhyphen} command may be redefined to do nothing if
this behaviour is not wanted. (It is defined by \app{texjavahelpmk}
but has no function.) 

\section{Document Division Commands}
\label{sec:docdivisioncmds}

The standard \LaTeX\ sectioning commands \inlineglsdef{part}, \inlineglsdef{chapter},
\inlineglsdef{section}, \inlineglsdef{subsection},
\inlineglsdef{subsubsection}, \inlineglsdef{paragraph} and
\inlineglsdef{subparagraph} can be used as
normal, but they should all be followed by
\code{\gls{label}\margm{label}}, which allows the section to be
referenced by a context-sensitive help button. The label also
determines the file basename if the section should be split into a
separate file. Any colon prefix (such as \qtt{ch:} or \qtt{sec:})
will be stripped and problematic characters will be replaced with underscores when
forming the basename.

For example, this section is started with:
\begin{codebox}
\gls{section}\marg{Document Division Commands}
\gls{label}\marg{sec:docdivisioncmds}
\end{codebox}
So the section label is \code{sec:docdivisioncmds} but the
\qtt{sec:} prefix will be stripped when determining the filename.
This means that the filename for this section has the basename
\filefmt{docdivisioncmds} (the file extension is determined by
\app{texjavahelpmk}['s] \TJHMKswitch{html-ext} option, and defaults
to \ext{html}).

\begin{information}
If any glossary entry hyperlinks where the target is in the form
\code{wrglossary.}\meta{n} have the wrong filename in the link,
use the \optval{debug}{showwrglos} option when loading
\sty{texjavahelp}. (This will be passed to \sty{glossaries-extra}.)
Then compare the target numbers in the PDF with those in the HTML
files. The problem may stem from an entry being indexed when the
document is built with \LaTeX\ but \app{texjavahelpmk} omits the
indexing.
\end{information}

The navigation tree is determined by the table of contents \ext{toc}
file. However, there are some divisions that are not typically
included in the table of contents (such as the table of contents
itself) that need to be in the navigation tree.
Some commands are provided to ensure that nodes are added to the
navigation tree where there is no corresponding entry in the
\ext{toc} file.

\cmddef{WriteTJHDivisionInfo}
This simply writes a line to the \ext{aux} file, which
\app{texjavahelpmk} can pick up. This should be placed immediately
after the section heading. For example:
\begin{codebox}
\gls{chapter}*\marg{Foreword}
\gls{WriteTJHDivisionInfo}\marg{chapter}\marg{}\marg{Foreword}\marg{foreword}\marg{\cmd{thepage}}
\end{codebox}
The \meta{unit} argument should be the sectional unit
(\qtt{chapter}, \qtt{section} etc). The \meta{prefix} argument is
the title prefix, if applicable. This should be empty if there's no
prefix (which is more likely to be the case as the prefix is
typically the chapter or section number). The \meta{target} is a
label that will be used as the basis of the node filename.
The \meta{location} argument will typically be the page number, but
it's mostly ignored.

If \cls{scrbook} or \cls{scrreport} have been loaded, this command
will be used after the table of contents title. It's automatically added with:
\begin{compactcodebox}
\cmd{AfterTOCHead}\oarg{toc}
 \marg{\gls{WriteTJHDivisionInfo}\marg{chapter}\marg{}\marg{\cmd{contentsname}}\marg{toc}\marg{\cmd{thepage}}}
\end{compactcodebox}

Any sectioning units that are added to the table of
contents will need to have a label after the chapter heading
to set the corresponding file's basename.  This includes unnumbered
chapters that are automatically added with KOMA-Script's
\optfmt{toc=listof} option. For example:
\begin{compactcodebox}
\cmd{AfterTOCHead}\oarg{lof}\marg{\cmd{label}\marg{listoffigures}}
\cmd{AfterTOCHead}\oarg{lot}\marg{\cmd{label}\marg{listoftables}}
\end{compactcodebox}
This doesn't need to be done for navigational units added with 
\gls{WriteTJHDivisionInfo} as the \meta{target} argument already specifies
this.

\cmddef{HeadlessSection}
This is a shortcut command that creates a hyperlink target and uses
\gls{WriteTJHDivisionInfo} where a new navigation node should start
but there's no associated title. The \meta{title} argument is the
title for the navigation node and does not appear in the document.
The other arguments are as for \gls{WriteTJHDivisionInfo}.

\envdef{prefrontmatterblock}
A shortcut environment that's only available if \gls{chapter} is
defined. This simply does:
\begin{compactcodebox*}
\gls{clearpage}
\gls{HeadlessSection}\margm{target}\marg{chapter}\margm{title}
\gls{pagestyle}\marg{empty}
\meta{body}
\gls{clearpage}
\end{compactcodebox*}

For example:
\begin{codebox}
\cbeg{prefrontmatterblock}\marg{note}\marg{Note}
Some untitled front matter note on a page with no header or footer.
\cend{prefrontmatterblock}
\end{codebox}

\cmddef{WriteTJHLabel}
This is used to establish where a label should actually be for cases
where the label, created with \gls{label}, might float to a different division. The
floating commands \gls{FloatFig}, \gls{FloatSubFigs} and \gls{FloatTable}
all use this command to counteract the possibility that the floats
may end up in a different section. If you directly use floating
environments you may find that the HTML anchor corresponding to the
label is associated with the wrong HTML file. If this happens, you
will need to place \gls{WriteTJHLabel} at the place where the float
(which won't float in the HTML file) will occur in the HTML file.

\begin{important}
\gls{WriteTJHLabel} must be used outside of the floating
environment. The actual label must still be created with
\gls{label}.
\end{important}

For example:
\begin{codebox}
\gls{WriteTJHLabel}\marg{fig:example}
\cbeg{figure}[p]
\gls{includegraphics}\marg{example-image}
\gls{caption}\marg{Example Image}
\gls{label}\marg{fig:example}
\cend{figure}
\end{codebox}

\section{Floats}
\label{sec:floats}

Floating environments, such as \env{figure} or \env{table}, can
cause \app{texjavahelpmk}['s] referencing system to fail if they
float to a different section. The associated anchor may end up being
associated with the wrong HTML filename.

The \sty{texjavahelp} package provides some commands to help correct
anchors and provide HTML code that can be more easily adjusted via
CSS styling.

\cmddef{FloatFig}
Creates a \env{figure} environment with the given content centred
and followed by the provided caption and label. The \meta{pos}
argument, if set, will be provided as the \env{figure} option
argument. The \meta{lof} optional argument is for the list of
figures. If present, it will be inserted as an optional argument to
\gls{caption}. The \meta{content} is the figure content, typically
an image included with \gls{includegraphics} or \gls{includeimg}.
The \meta{caption} argument is the caption text and \meta{label}
is the caption label.

The \meta{\LaTeX\ only pre-content} is added immediately
before the \meta{content} (inside the \env{figure}) but only for the
PDF version.
The \meta{\LaTeX\ only pre-content} should not contain anything that would
upset the indexing or cross-references. It's mainly intended to
allow changes to the \sty{graphicx} settings just for the PDF.
It will be ignored by \app{texjavahelpmk}. For example:
\begin{codebox}
\gls{FloatFig}\oarg{p}\marg{fig:sample}
 \oarg{\cmd{setkeys}\marg{Gin}\marg{width=\cmd{textwidth}}}
 \marg{\gls{includegraphics}\marg{example-image}}
 \marg{An Example Image}
\end{codebox}
When compiling the document normally, this essentially does:
\begin{compactcodebox}
\gls{WriteTJHLabel}\marg{fig:sample}
\cbeg{figure}\oarg{p}
 \cmd{centering}
 \cmd{setkeys}\marg{Gin}\marg{width=\cmd{textwidth}}
 \gls{includegraphics}\marg{example-image}
 \gls{caption}\marg{An Example Image}
 \gls{label}\marg{fig:sample}
\cend{figure} 
\end{compactcodebox}
However, \app{texjavahelpmk} will create a \xmltagfmt{div} element
with the \xmlattr{id} attribute set to \code{fig:sample}.
(The colon will be stripped if the XML setting is on.)
This should mean that hyperlinks to the figure will scroll to the
top of the figure rather than to the caption (which would cause the
figure content to scroll out of sight).

\cmddef{FloatSideBySideFigs}
Creates two side-by-side figures within a single \env{figure}
environment. Each figure is encapsulated within a \env{minipage}
environment with the width calculated as half the text width less
the side-by-side separator dimension. The content of each
\env{minipage} is centred and followed by the corresponding caption and label.

The \meta{pos} optional argument, if not empty, is the optional
argument to pass to the \env{figure} environment. 
The \meta{\LaTeX\ only pre-both} argument is placed after the start of the
\env{figure} environment and before the first \env{minipage}. 
This argument is ignored by \app{texjavahelpmk} and may be used to
adjust the \sty{graphicx} parameters just for the PDF.

The \meta{caption1} and \meta{caption2} arguments are the text for
the first and second captions. If different text is required for the
list of figures, this can be specified with the optional
\meta{lof1} and \meta{lof2} arguments.
The \meta{label1} and \meta{label2} arguments are the label for the
first and second caption, respectively.

The content for the first figure is specified with the
\meta{content1} argument. If the optional \meta{\LaTeX\ only pre-content1} 
argument is provided, this will be placed immediately before
\meta{content1} but will be ignored by \app{texjavahelpmk}.

The content for the second figure is specified with the
\meta{content2} argument. If the optional \meta{\LaTeX\ only pre-content2} 
argument is provided, this will be placed immediately before
\meta{content2} but will be ignored by \app{texjavahelpmk}.

The \meta{\LaTeX\ pre-figure content\slash HTML between figures}
argument is placed before the start of the \env{figure} environment
for the PDF and between the two figures for the HTML. 
This allows the PDF to be more compact.

\cmddef{floatsidebysidevalign}
The vertical alignment for the two \env{minipage} environments in
\gls{FloatSideBySideFigs} is obtained by expanding
\gls{floatsidebysidevalign}.

\cmddef{FloatSubFigs}
Creates a figure with sub-figures as the content. The starred
version puts the first sub-figure on a line of its own. The
sub-figures will all be placed inside the \env{figure} environment.
The optional \meta{pos} argument, if present, will be passed as the
optional argument to the \env{figure} environment.

The \meta{caption} argument is the caption text for the entire figure.
If alternative text is required for the list of figures, this can be
specified with the optional \meta{lof} argument. The \meta{label}
argument is the label for the caption of the entire figure.

Within the \env{figure} environment the \ctr{subfigure} counter will
be reset to 0 and the content will be centred. The \meta{\LaTeX\
only pre-content} will be inserted before the sub-figures, but only
for the PDF. This argument will be ignored by \app{texjavahelpmk}.
It can be used, for example, to locally change the \sty{graphicx}
parameters just for the PDF.

The \meta{sub-figure list} argument should be a comma-separated list
where each item in the list must be in the form
\begin{compactcodebox}
\margm{label}\margm{content}\oargm{hyperref title}\margm{caption}
\end{compactcodebox}
where \meta{label} is the sub-figure label, 
\meta{content} is the sub-figure content, and
\meta{caption} is the caption text (which may be empty).
If the \meta{hyperref title} optional argument is present, 
\gls{@currentlabelname} will be set to \meta{hyperref title}
otherwise it will be set to \meta{caption}.

For example:
\begin{codebox}
\gls{FloatSubFigs}\marg{fig:samplesubfigs}
\oarg{\cmd{setkeys}\marg{Gin}\marg{width=0.45\cmd{linewidth}}}
\marg{
 \marg{fig:samplesubfigsa}\marg{\gls{includeimg}\marg{imagea}}\marg{},
 \marg{fig:samplesubfigsb}\marg{\gls{includeimg}\marg{imageb}}\marg{}
}
\oarg{An Example Figure With Sub-Figures}
\marg{An Example Figure With Sub-Figures:
\gls{subfigref}\marg{fig:samplesubfigsa} first sub-figure,
\gls{subfigref}\marg{fig:samplesubfigsb} second sub-figure.}
\end{codebox}

Each sub-figure is formatted with:
\cmddef{SubFigureContent}
where the arguments are obtained from the corresponding item in the
\meta{sub-figure list} argument of \gls{FloatSubFigs}.
This uses a \env{tabular} environment to centre the sub-figure image content
above the caption. This command is also responsible for incrementing
the \ctr{subfigure} counter.

A sub-figure can be referenced with \gls{ref} (or \gls{figureref}
etc) but there it will include the parent figure number. If you only
want the actual sub-figure reference (which by default is a
lowercase letter: a, b, etc) then you can use:
\cmddef{subfigref}
This is like \gls{ref} but omits the parent figure number prefix.
The sub-figure reference is formatted with:
\cmddef{subfigurefmt}
This simply does \code{(\cmd{emph}\margm{text})}.

\cmddef{FloatTable}
This command is analogous to \gls{FloatFig} but places its content
inside a \env{table} environment. In this case the content would
typically be a \env{tabular} environment. Again, the purpose of this
command, rather than explicitly using the \env{table} environment,
is to assist \app{texjavahelpmk} in creating cleaner HTML code
(as well as using \gls{WriteTJHLabel} in case the table floats
outside of the current section).

The optional \meta{pos} argument, if present, will be passed as the
optional argument to the \env{table} environment. The \meta{\LaTeX\
only pre-content} is placed at the start of the \env{table}
environment and may be used to make localised adjustments, for
example, to change the value of \gls{tabcolsep}. This argument is
ignored by \app{texjavahelpmk}.
The \meta{caption} argument is the caption text. If the \meta{lot}
argument is present, this will be used for the list of tables
instead of the \meta{caption}. The \meta{label} argument is the
caption label.

If you have a long table that needs to span multiple pages, you can
instead use:
\cmddef{NoFloatLongTable}
You will need to load the \sty{longtable} package if you want to use
this command as it will use the \env{longtable} environment.
The optional \meta{h-align} argument, if present, will be added as
the optional argument to the \env{longtable} environment.
The \meta{label}, \meta{\LaTeX\ only pre-content}, \meta{caption}
and \meta{lot} arguments are as for \gls{FloatTable}.

The \meta{content} argument is different from the corresponding
argument in \gls{FloatTable} as now it shouldn't include the
\env{tabular} environment. Instead, the column specifiers should be
provided in the \meta{col-specs} argument, the header row should be
provided in the \meta{header row} argument and the table body should
be in the \meta{content} argument.

\gls{NoFloatLongTable} will automatically set the \env{longtable}
head and foot. Each foot, except for the last, will have
\qt{\continuednextpage}. This is encapsulated with:
\cmddef{ContinuedNextPageBox}
This uses \gls{multicolumn} to span all columns, with the \code{r}
alignment specifier and the content is \meta{text} encapsulated
with:
\cmddef{continuedfmt}
This simply does \code{\cmd{emph}\margm{text}}.

Each head, apart from the first, will have \qt{\continuedname} appended
to the caption.  Again, this is encapsulated with
\gls{continuedfmt}.

If you need to stack text vertically within a table cell to make it
more compact for the PDF, you can use:
\cmddef{tablecellstack}
This is simply defined to do \code{\cmd{shortstack}\margm{content}}
by the \sty{texjavahelp} package, but with \app{texjavahelpmk} the
new row \gls{dbbksl} markup will be replaced with a space.

Some convenient commands are provided for cross-referencing.
\cmddef{figureref}
Used to reference figures, this is a shortcut command that uses
\gls{ref} or \starredcs{ref} and is prefixed by:
\cmddef{figurerefprefix}
This simply expands to \gls{figurename} followed by a non-breakable
space.

\cmddef{Figureref}
Used to reference figures, this is a shortcut command that uses
\gls{ref} or \starredcs{ref} and is prefixed by:
\cmddef{Figurerefprefix}
This simply expands to \gls{Figurename} followed by a non-breakable
space.

If you want to reference a list of figures, you can instead use:
\cmddef{figuresref}
where the argument is a comma-separated list of labels.
In this case the prefix is given by:
\cmddef{figuresrefprefix}
This simply expands to \gls{figuresname} followed by a non-breakable
space.

There is also a sentence-case version:
\cmddef{Figuresref}
where the argument is a comma-separated list of labels.
In this case the prefix is given by:
\cmddef{Figuresrefprefix}
This simply expands to \gls{Figuresname} followed by a non-breakable
space.

There are analogous commands for referencing tables.
\cmddef{tableref}
This shortcut command uses \gls{ref} or \starredcs{ref} and is prefixed by:
\cmddef{tablerefprefix}
This simply expands to \gls{tablename} followed by a non-breakable
space.

\cmddef{Tableref}
This shortcut command uses \gls{ref} or \starredcs{ref} and is prefixed by:
\cmddef{Tablerefprefix}
This simply expands to \gls{Tablename} followed by a non-breakable
space.

If you want to reference a list of tables, you can instead use:
\cmddef{tablesref}
where the argument is a comma-separated list of labels.
In this case the prefix is given by:
\cmddef{tablesrefprefix}
This simply expands to \gls{tablesname} followed by a non-breakable
space.

There is also a sentence-case version:
\cmddef{Tablesref}
where the argument is a comma-separated list of labels.
In this case the prefix is given by:
\cmddef{Tablesrefprefix}
This simply expands to \gls{Tablesname} followed by a non-breakable
space.


\section{Image Commands}
\label{sec:imagecmds}

The \sty{graphicx} package is automatically loaded so you can use
\gls{includegraphics} (the definition with the \keyvallist\ optional
argument). The \gls{TeXParserLib} supports this command (but not
all options). The \sty{tikz} package is also automatically loaded
but is not supported by the \gls{TeXParserLib}.

\cmddef{TeXParserLibToImage}
This just does \meta{\LaTeX\ code} when compiling with
\LaTeX\ but the \gls{TeXParserLib} will attempt to convert
the supplied code to an image if \app{texjavahelpmk} is run 
with the default \TJHMKswitch{convert-images} setting.
Recognised options:

\optiondef{TeXParserLibToImage.alt}
Sets the alt text. If not supplied, the \LaTeX\ code will be
used.

\optiondef{TeXParserLibToImage.pre-alt}
If set, inserts \meta{text} at the start of the alt text.

\optiondef{TeXParserLibToImage.post-alt}
If set, inserts \meta{text} at the end of the alt text.

\optiondef{TeXParserLibToImage.name}
Set the image file basename to \meta{basename}. If not
set, a generic basename is used in the form \metafilefmt{img}{n}{}
(for example, \filefmt{img000001.png}). The file extension is
determined by the image type.

\optiondef{TeXParserLibToImage.type}
Identifies the type of image to create, where
\meta{mime-type} should be a supported image MIME type
(\code{image/png}, \code{image/jpeg} or \code{application/pdf}).
In the case of PDF, the image will be embedded in the HTML file
using the \code{object} element. Note that this is not supported
by HTMLDocument. However, a PNG file will be created to use as a
fallback.

\optiondef{TeXParserLibToImage.crop}
A boolean option that determines whether or not to crop the image.

\optiondef{TeXParserLibToImage.div}
Indicates whether or not to encapsulate the image in a
\code{div} element. The value may be boolean (\optfmt{true} or
\optfmt{false}) or the CSS class name.

\cmddef{TeXParserLibToImagePreamble}
If defined, the value of this token list variable will 
be added to the document preamble used to create the image for
\gls{TeXParserLibToImage}.

\cmddef{includeimg}
Behaves like \gls{includegraphics} if \metafilefmt{}{basename}{.tex}
does not exist, otherwise \gls{includeimg} sets the input path to
the same as for \gls{includegraphics} and inputs
\metafilefmt{}{basename}{.tex} (where the file is expected to
include \LaTeX\ image code, such as \sty{tikz} commands).
With \app{texjavahelpmk}, a bitmap will be searched for with the
same basename, otherwise
it the default \TJHMKswitch{convert-images} setting is on,
\app{texjavahelpmk} will convert the image to a bitmap.

\cmddef{includetimg}
As \gls{includeimg} but top-aligns the image.

The options for \gls{includeimg} and \gls{includetimg} are a subset
of those for \gls{includegraphics}: \includeimgopt{scale},
\includeimgopt{width}, \includeimgopt{height},
\includeimgopt{rotate}, \includeimgopt{alt},
\includeimgopt{actualtext}, and \includeimgopt{artifact}.
Additionally, the \includeimgopt{frame} boolean option is also
available. If true, this will put a frame around the image.

\section{Entry Referencing Commands}
\label{sec:tjhglscmds}

The \sty{glossaries-extra-bib2gls} package provides some commands
for use with labels that may have a prefix from a known set.
Once you have established the set of label prefixes, you can use
commands like \gls{dgls} or \gls{dglsfield} to reference entries
without including the prefix.

The \sty{texjavahelp} package sets up the label prefix list with the
following:
\begin{compactcodebox}
\gls{glsxtraddlabelprefix}\marg{action.}
\gls{glsxtraddlabelprefix}\marg{button.}
\gls{glsxtraddlabelprefix}\marg{menu.}
\gls{glsxtraddlabelprefix}\marg{widget.}
\gls{glsxtraddlabelprefix}\marg{help.}
\gls{glsxtraddlabelprefix}\marg{index.}
\gls{glsxtraddlabelprefix}\marg{label.}
\gls{glsxtraddlabelprefix}\marg{}
\end{compactcodebox}

This means that you can use \gls{dgls} for entries with any of these
prefixes, for example, \code{\gls{dgls}\marg{help}} would match
\qtt{button.help} so it would be equivalent to
\code{\gls{gls}\marg{button.help}} but it wouldn't pick up
\code{\gls{gls}\marg{menu.help}} which might also be valid.

The following commands are provided by the \sty{texjavahelp} package, some
of which use this label prefix system, but some use a specific
prefix instead.

\cmddef{widget}
Similar to \gls{glstext} but encapsulates the \bibentryfield{text}
field in \gls{widgetfmt} and uses the prefix label list
to find the first valid prefix. There are case-changing
alternatives, \inlineglsdef{Widget} (sentence-case) and
\inlineglsdef{WIDGET} (all-caps) although widget text is
more typically formatted using the same case as shown in the application.
For example:
\begin{coderesult}
\gls{widget}\marg{help}
\tcblower
\widget{help}
\end{coderesult}

\cmddef{btn}
Similar to \gls{dgls} (with no modifier) but this command will check
if the \bibentryfield{iconimage} field has been set. If it has, 
the image will be inserted before the text, separated by:
\cmddef{btniconsep}
This expands to \csfmt{space} by default. For example:
\begin{coderesult}
\gls{btn}\marg{help}
\tcblower
\btn{help}
\end{coderesult}
Compare this with the earlier \code{\gls{widget}\marg{help}}
example, which doesn't show the icon image. The image file will
either need to be on the graphics path so that \gls{includegraphics} can find
it, or in a PDF file created with \app{tjhcreateiconpdf} where the
image mappings are loaded with \gls{TJHRequireIcons}.

The frame around the icon images is added by \gls{TJHLargeIconImageFrame}
or \gls{TJHSmallIconImageFrame}, depending on whether
\gls{tjhLargeIcon} or \gls{tjhSmallIcon} was used to show the icon
image. Note that \gls{TJHSmallIconImageFrame} not only puts a frame
around the image, but will also resize the image if the definition of
\gls{tjhsmalliconheight} is not empty.

These frame commands may be redefined but they are not used by \app{texjavahelpmk}.
Instead the CSS style determines whether or not any border should be
added. However, bear in mind Java's limited CSS support.

\cmddef{menutrail}
Displays the menu trail that leads to the given entry (identified by
\meta{entry-label}), which represents a menu item. The trail is 
obtained from the entry's hierarchy but it stops if a parent doesn't
have the \bibentryfield{category} set to \qtt{menu}. Each item is
formatted using \gls{menuitemref}.
For example:
\begin{coderesult}
\gls{menutrail}\marg{menu.help.manual}
\tcblower
\menutrail{menu.help.manual}
\end{coderesult}

\cmddef{menusep}
Expands to the separator used by \gls{menutrail}.

\cmddef{menuitemref}
Formats the entry item in \gls{menutrail}. This just expands to
\code{\gls{glshyperlink}\margm{label}}.

\cmddef{menu}
This first does \code{\gls{menutrail}\margm{entry-label}} and
then does \code{\gls{glsadd}\oargm{options}\margm{entry-label}} 
to index the entry. The \meta{entry-label}
is formed by concatenating \meta{label-prefix} and \meta{label}.
The \meta{prefix-label} defaults to \qtt{menu.}\ and
the starred version prepends \glsoptval{format}{\glscsname{glsignore}} to
\meta{options}.
For example:
\begin{coderesult}
\gls{menu}\marg{help.manual}
\tcblower
\menu{help.manual}
\end{coderesult}

\cmddef{keyref}
A shortcut for:
\begin{compactcodebox}
\gls{gls}\oargm{options}\marg{manual.keystroke.\meta{label}}\oargm{insert}
\end{compactcodebox}
This command may be used to reference keystrokes but they won't
appear in the index as the \qtt{manual.}\ prefix will cause them to
be moved into an ignored glossary.
For example:
\begin{coderesult}
\gls{keyref}\marg{pageup}
\tcblower
\keyref{pageup}
\end{coderesult}
There's no hyperlink because there's no target.

\cmddef{keydescref}
A shortcut that uses \gls{gls} to reference the entry with the
label \code{manual.keystroke.\meta{label}} and the \glsopt{format}
set to \glscsname{keys} followed by
(if set) the description converted to lowercase. This command doesn't have modifiers.
For example:
\begin{coderesult}
\gls{keydescref}\marg{pageup}
\tcblower
\keydescref{pageup}
\end{coderesult}
Note the difference in the way the keystroke is formatted compared
with just using \gls{keyref}.

\cmddef{desckeyref}
Similar to \gls{keydescref} but the description (if set) comes
first.
\begin{coderesult}
\gls{desckeyref}\marg{pageup}
\tcblower
\desckeyref{pageup}
\end{coderesult}

\cmddef{Desckeyref}
As \gls{desckeyref} but if the description is present, it will be
converted to sentence-case. No case-change will be applied if the
description isn't present.
\begin{coderesult}
\gls{Desckeyref}\marg{pageup}
\tcblower
\Desckeyref{pageup}
\end{coderesult}
Note that in this case, the description is actually specified in
title case:
\begin{compactcodebox}
\bibentry{keystroke}\marg{manual.keystroke.pageup,
  \bibentryfield{name}=\marg{PgUp},
  \bibentryfield{description}=\marg{Page Up}
}
\end{compactcodebox}
This means that the case change has no effect.



\chapter{The \apptext{texjavahelpmk} Command Line Application}
\label{sec:texjavahelpmk}

\appdef{texjavahelpmk}

Use the \TJHMKswitch{help} switch 
(or \swch{help}) for help and the
\TJHMKswitch{version} switch
(or \swch{version}) for the version information.

\TJHMKswitchdef{help}
Display help message and exit.

\TJHMKswitchdef{version}
Display version information and exit.

\TJHMKswitchdef{verbose}
Switch on verbose mode.

\TJHMKswitchdef{noverbose}
Switch off verbose mode.

\TJHMKswitchdef{in}
Specifies the input file. The switch may be omitted (that is,
\meta{tex-file} may just be provided) provided that
\TJHMKswitch{output} is also used or the input file is specified before
the output directory.

\TJHMKswitchdef{output}
Specifies the output directory. The switch may be omitted
(that is, \meta{out-dir} may just be provided) provided that
\TJHMKswitch{in} is also used or the output directory is specified after
the input file.

\TJHMKswitchdef{html-ext}
The extension to use for the HTML files.

\TJHMKswitchdef{split}
Specifies the sectioning level to split on. The level should be a
number that corresponds to the section level (0 for chapter, 1 for
section, etc).

Each section should be labelled as the HTML filenames
are constructed from the label. Any colon prefix such as
\code{sec:}\ or \code{ch:}\ will be stripped and any periods will be
removed or replaced by underscores. If a section isn't labelled, the basename
will be \metafilefmt{node}{n}{} instead.

\TJHMKswitchdef{prefix-split}
The HTML filename will be prefixed with the input file basename.
For example, if the input file is \filefmt{foo.tex} and the
introduction section has been assigned the label \code{sec:intro}
then the HTML file for the introduction section will be
\filefmt{foo-intro.html} instead of just \filefmt{intro.html}.

\TJHMKswitchdef{noprefix-split}
Don't prefix the HTML filename with the input file basename.

\TJHMKswitchdef{charset}
Sets the default character encoding.

\TJHMKswitchdef{out-charset}
Sets the character encoding for the output files.

\TJHMKswitchdef{image-dest}
Specifies where to put output images. The given directory should be
relative to the output directory.  If \meta{directory} is an empty
string (the default) then the images will be copied to a location
relative to the output directory that retains the same structure as
for the input. 

For example, if the input file \filefmt{doc.tex}
is in the directory \filefmt{doc/} and the image \filefmt{img1.png} is in
\filefmt{doc/images1/} and the image \filefmt{img2.png} is in
\filefmt{doc/images2/} and, if the output directory is
\filefmt{helpset/}, then with the default setting, \filefmt{img1.png}
will be copied to \filefmt{helpset/images1/} and \filefmt{img2.png}
will be copied to \filefmt{helpset/images2/}. However, if
\TJHMKswitch{image-dest} sets the image destination to \filefmt{images}
then both \filefmt{img1.png} and \filefmt{img2.png} will be copied
to \filefmt{helpset/images/}.  

\TJHMKswitchdef{convert-images}
Use \pdfLaTeX\ or \LuaLaTeX\ to convert \LaTeX\ code (with
\gls{TeXParserLibToImage} or input with \gls{includeimg} etc) to
bitmap images. This is the default. The temporary files used in this
operation will be deleted by default. Use \TJHMKswitch{no-rm-tmp-dir}
to retain them.

\TJHMKswitchdef{noconvert-images}
Don't convert \LaTeX\ image files.

\TJHMKswitchdef{image-preamble}
If an image needs to be automatically generated, the temporary
\LaTeX\ file used to create the image should have the preamble set
to \meta{\LaTeX\ code}. If not set, the preamble will be the same as
the preamble from the input file.

\TJHMKswitchdef{image-preamble-from-file}
As \TJHMKswitch{image-preamble} but the content is obtained from the file
identified by \meta{filename}.

The following options govern the HTML output.

\TJHMKswitchdef{entities}
Use HTML entities for non-ASCII characters. This will interfere with
the helpset search data.

\TJHMKswitchdef{noentities}
Don't use HTML entities for non-ASCII characters (default).

\TJHMKswitchdef{support-unicode-script}
Use Unicode superscripts or subscripts where available.

\TJHMKswitchdef{nosupport-unicode-script}
Don't use Unicode superscripts or subscripts, just
use \code{sub} or \code{sup} elements (default).

\TJHMKswitchdef{head}
Add \meta{html} code to the HTML header.

\TJHMKswitchdef{body-preamble}
Insert \meta{html} at the start of the HTML body.

\TJHMKswitchdef{body-preamble-from-file}
As \TJHMKswitch{body-preamble} but the content is obtained from the file
identified by \meta{filename}.

\TJHMKswitchdef{body-postamble}
Insert \meta{html} at the end of the HTML body.

\TJHMKswitchdef{body-preamble-from-file}
As \TJHMKswitch{body-preamble} but the content is obtained from the file
identified by \meta{filename}.

\TJHMKswitchdef{helpset}
Create helpset files. These files are designed for integration with
the \gls{TeXJavaHelpLib}, which means that there won't be any
HTML5 markup (as it's not supported by Java's \code{HTMLDocument} class).
There also won't be any navigation added as that's dealt with by the
Help window (see \sectionref{sec:helpwindows}).
This setting automatically implements
\TJHMKswitch{nobreadcrumbtrail}, \TJHMKswitch{nominitoc} and
\TJHMKswitch{nomathjax}.

\TJHMKswitchdef{html}
Indicates that the HTML files are not intended for use with the
\gls{TeXJavaHelpLib}, so HTML5 markup will be used and a
navigation line will be added to the end of each HTML body. This setting
also automatically implements \TJHMKswitch{breadcrumbtrail},
\TJHMKswitch{minitoc} and \TJHMKswitch{mathjax}.

\TJHMKswitchdef{nohelpset}
Deprecated synonym of \TJHMKswitch{html}.

\TJHMKswitchdef{epub}
This is still experimental. This switch indicates that an EPUB file
should be created. Since this format requires XHTML, this setting
will switch on \app{texjavahelpmk}['s] XML mode. All the HTML and
CSS files will be created in the target directory (but not the XML
files that would ordinarily be created with \TJHMKswitch{helpset}) and
also the additional files required by the EPUB format. The EPUB file
will be created using these files (but it will also contain the
\filefmt{mimetype} file which won't be created separately). 

This option automatically implements \TJHMKswitch{nomathjax},
\TJHMKswitch{nominitoc} and \TJHMKswitch{nobreadcrumbtrail}.

\begin{important}
The XML format has restrictions on the characters permitted as
identifiers so \app{texjavahelpmk} will strip or replace problematic
characters in labels but this isn't guaranteed to always work.
\end{important}

\TJHMKswitchdef{mathjax}
Add MathJax support. Note that Java's \code{HTMLDocument} class
doesn't support MathJax, so this setting is off by default.

\TJHMKswitchdef{nomathjax}
Don't add MathJax support. This will just use an HTML
approximation which is quite limited.

\TJHMKswitchdef{breadcrumbtrail}
Add breadcrumb trail to the start of HTML body.

\TJHMKswitchdef{nobreadcrumbtrail}
Don't add breadcrumb trail to the start of HTML body.

\TJHMKswitchdef{minitoc}
Add minitoc to HTML body. This isn't required for helpset files as
the Help window provides a navigation panel.

\TJHMKswitchdef{nominitoc}
Don't add minitoc to HTML body.

\TJHMKswitchdef{minitoc-div-class}
If \meta{class-name} not empty, encapsulate minitoc in a
div with the given class name. This option is ignored with
\TJHMKswitch{nominitoc}.

\TJHMKswitchdef{minitoc-div-id}
If \meta{id-name} not empty, encapsulate minitoc in a
div with the id class name. This option is ignored with
\TJHMKswitch{nominitoc}.

If neither \TJHMKswitch{minitoc-div-class} nor \TJHMKswitch{minitoc-div-id} is
set to a non-empty value, then the encapsulating div won't be added.

\TJHMKswitchdef{minitoc-preamble}
Insert \meta{html} at the start of the minitoc (inside the
encapsulating div if \TJHMKswitch{minitoc-div-class} or \TJHMKswitch{minitoc-div-id} have been
set).

\TJHMKswitchdef{minitoc-preamble-from-file}
As \TJHMKswitch{minitoc-preamble} but the content is obtained from the file
identified by \meta{filename}.

\TJHMKswitchdef{minitoc-postamble}
Insert \meta{html} at the end of the minitoc (inside the
encapsulating div if \TJHMKswitch{minitoc-div-class} or \TJHMKswitch{minitoc-div-id} have been
set).

\TJHMKswitchdef{minitoc-postamble-from-file}
As \TJHMKswitch{minitoc-postamble} but the content is obtained from the file
identified by \meta{filename}.

The following options are provided for debugging purposes.

\TJHMKswitchdef{no-rm-tmp-dir}
Don't remove temporary files on exit.

\TJHMKswitchdef{rm-tmp-dir}
Remove temporary files on exit (default).

\TJHMKswitchdef{log}
Set the \TeX\ Parser log file to \meta{filename}.

\TJHMKswitchdef{nolog}
Don't log messages.

\TJHMKswitchdef{debug}
Switch on debug mode. The \meta{level} (if present) should be an
integer greater than 0. (Setting the level to 0 is equivalent to
switching off debugging.)

\TJHMKswitchdef{debug-mode}

Enable \TeX\ Parser debugging. Be aware that this can be extremely
verbose.

\TJHMKswitchdef{nodebug}
Switch off debugging mode (default).

\TJHMKswitchdef{no-debug}
Synonym for \TJHMKswitch{nodebug}.

\chapter{The \TeXJavaHelp\ Library (\filetext{texjavahelplib.jar})}
\label{sec:texjavahelplib}

The \inlineglsdef{file.texjavahelplib.jar} file contains the 
\glsxtrfull{TeXJavaHelpLib} classes that may be used to load the
files created by \app{texjavahelpmk} and display the information in
a frame with navigation buttons. The library also deals with loading
localisation files, so \app{tjhxml2bib}, \app{texjavahelpmk} and the
application requiring the \gls{TeXJavaHelpLib}'s \gls{gui}
elements need to have \file{texjavahelplib.jar} on the class path.

The library also comes with images that are used as icons in the
help frame buttons and \glspl{menu}. Text and icons for common buttons 
\widget+{okay}, \widget+{apply}, \widget+{close} and
\widget+{cancel} are provided, but may be overridden by
providing alternative values in the application language file
or in the application's icon directory.

The keyword search will only work if \app{texjavahelpmk} has created
the search \gls{xml} file, which catalogues the search terms with a
reference to the file and block or paragraph where the terms were
found. Similarly, the index is created during the document build
process, which depends on \sty{glossaries-extra} and \app{bib2gls}
to create the required files that are parsed by \app{texjavahelpmk}.

The file \filefmt{helpinterface.tex} provides a description of the
help system and is designed to be input into the application's
documentation source file (as in \sectionref{sec:helpwindows}).

The main help frame can't be accessed when a modal dialog is open.
There is therefore an interface to allow just the pages applicable
to a dialog to be viewed. This means that a help button in the
dialog box can open a minimal view. If the main help frame happens
to be open at the same time, it will be closed when the help dialog
is opened to avoid confusion. The help dialog's page will be added
to the main help frame's history so that the user can go back to it
after the dialog has been closed by simply reopening the main help
frame.
