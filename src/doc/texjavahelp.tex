\begin{prefrontmatterblock}{note}{Note}
The \TeX\ Java Help System is provided \qt{as is}. It's limited to a
few supported packages. It's designed to integrate into a Java
application's in-application manual and context-sensitive help
buttons (as well as providing a PDF for general use). 
Since Java's HTML and CSS support is quite limited, this
means that the files generated by the \TeX\ Java Help System must be
able to comply with this limitation.

\begin{warning}
This manual is incomplete and the library is still a work in
progress.
\end{warning}

\end{prefrontmatterblock}

\chapter{Introduction}
\label{sec:intro}

I have a number of Java \gls{gui} applications, which require a manual.
Usually I create \gls{pdf} manuals using \pdfLaTeX\ or \LuaLaTeX, but
it's useful for a \gls{gui} to also have an in-application help, where 
an appropriate section can be opened for context-sensitive help (for example,
via a \widgetfmt{Help} button in a dialog).

In the past, I used \gls{JavaHelp} but it has a number of limitations and now
no longer seems to be supported. Ideally, I would like the same document source
to provide the offline \gls{pdf}, the \gls{html} and \gls{xml} files required
for the application's help system, and (if the documentation is particularly
large) \gls{html} files for the application's home page.

I've tried several ways to achieve this. The first was to use \LaTeX\ source
and convert to \gls{html}. The limitations of Swing's \varfmt{JEditorPane},
the complexity of adding support for \gls{JavaHelp}['s] custom
\xmltagfmt{object} format, and the need to create the supplementary \gls{xml}
files, meant that some additional work was required to get the correct output.

The second method I tried was to start with \gls{xml} source with a custom
script which generated both the \gls{html} and \gls{xml} files for \gls{JavaHelp}
and also the \LaTeX\ source for the \gls{pdf}. This proved the more reliable route
for the \gls{html} and \gls{xml} output, as it's much easier to convert from \gls{xml}
to \gls{html}, although I don't particularly like writing large documents in 
\gls{xml}.

One of the advantages of \gls{html} over \gls{pdf} is that it's more accessible.
This should mean that the in-application help ought to be more accessible than the
\gls{pdf} manual, but I couldn't find a way to allow the user to customise the
font without having to resort to editing the \gls{html} files (for example, to
make the font larger for visually impaired users or to change to a dyslexic font).

I'm now trying a different approach. Since most of my Java applications are in
some way related to \TeX, and I created the \gls{TeXParserLib} to parse
\LaTeX\ code to assist them, I decided to try using the \gls{TeXParserLib}
to parse the \LaTeX\ source code for the manual in order to create the 
\gls{html} and \gls{xml} files needed for a \gls{gui} help system. At the same
time, I decided to also create a library to provide a simple in-application help
system to replace \gls{JavaHelp}.

The biggest drawback is that the \gls{TeXParserLib} was originally designed to
perform limited parsing of documents: for extracting information, removing obsolete
or problematic code, or for converting short fragments into \gls{html}.
It wasn't intended for converting entire documents to \gls{html} and lacks support
for all but a handful of packages. The available support is essentially limited
to the support required for the various projects I've worked on. It's now 
principally geared towards \app{bib2gls} and \app{datatooltk}.

However, I have started using the \gls{TeXParserLib} to convert my \LaTeX\
package user manuals to \gls{html} to test the parsing capabilities. This means
that there's already support for my \sty{nlctuserguide} package. So I've started
by creating a new package, \sty{texjavahelp}, that's heavily based on
\sty{nlctuserguide} so that I can reuse much of the \gls{TeXParserLib}
support for \sty{nlctuserguide}. Unlike \sty{nlctuserguide}, which is 
designed for a self-contained source document, the \sty{texjavahelp} package
is designed to have accompanying \ext{bib} files for use with \app{bib2gls}.

The \TeX\ Java Help repository is at \url{https://github.com/nlct/texjavahelp}
and consists of:

\begin{itemize}
\item \sty{texjavahelp}: a \LaTeX\ package to help create the \gls+{pdf} version of
the documentation from the \LaTeX\ source. Requires the \sty{glossaries-extra}
package and \app{bib2gls}.

\item \app{tjhxml2bib}: a \gls{cli} application that converts
localisation resource files to \ext{bib} files suitable for use with
\app{bib2gls}.

\item \app{texjavahelpmk}: a \gls{cli} application that uses the \TeX\ Parser
Library (with a custom implementation of the \sty{texjavahelp} package) 
to create the \gls+{html} and \gls+{xml} files (required by the 
\TeXJavaHelp\ library) from the \LaTeX\ source.

\item \file{texjavahelplib.jar}: a Java library that provides a \gls{gui} 
component to display the \gls{html} files created by \app{texjavahelpmk}.

\item \app{texjavahelpdemo}: a \gls{gui} demo application to test 
\app{texjavahelpmk} and \file{texjavahelplib.jar}.
\end{itemize}

\begin{information}
It's not necessary to use the \sty{texjavahelp} package and \app{texjavahelpmk}.
The \gls{html} and \gls{xml} files can be created manually in a text editor 
without them, but they must be in the correct format. The use of \app{bib2gls}
for the glossary and index provides information written to the \file{index.xml}
file that's used in the tooltip text when hovering the mouse over a hyperlink
in the \gls{index.help-page}.
\end{information}

\chapter{The \stytext{texjavahelp} \LaTeX\ Package}
\label{sec:texjavahelpsty}

\pkgdef{texjavahelp}

The \sty{texjavahelp} package requires the following packages:
\sty{glossaries-extra}, \sty{fontawesome}, \sty{twemojis},
\sty{upquote}, \sty{hologo}, \sty{xcolor}, \sty{tcolorbox}, \sty{hyperref},
and \sty{tikz}. These are loaded automatically.
The \sty{glossaries-extra} package is loaded with the options:
\optval{record}{nameref} (which means that \app{bib2gls} should be used),
\opt{indexcounter}, \opt{floats}, \opt{nostyles}, 
\optvalm{stylemods}{tree\dcomma bookindex\dcomma topic} 
and \optval{style}{alttree}.

The following \sty{texjavahelp} package options are provided:
\optiondef{fontsymbols}
This option is the default, and defines \gls{tabsym} and \gls{upsym} in terms of
the font symbol commands \gls{barleftarrowrightarrowbar}
and \gls{baruparrow}. The \opt{fontsymbols} option will required a package that defines
these commands, such as \sty{stix} or \sty{boisk}, if those commands are required.

\optiondef{tikzsymbols}
This option  defines \gls{tabsym} and \gls{upsym} as images.
The \opt{tikzsymbols} option counteracts the \opt{fontsymbols} option.

Another other options supplied to \sty{texjavahelp} will be passed to
the \sty{glossaries-extra} package.

The \sty{texjavahelp} package is intended to be used with \app{bib2gls}. This means
that the document preamble should contain:
\cmddef{GlsXtrLoadResources}
This command is provided by the \sty{glossaries-extra} package and is used 
in conjunction with \app{bib2gls}. It writes information
to the \ext+{aux} file to be read by \app{bib2gls} and inputs
(if it exists) the \ext+{glstex} file created by \app{bib2gls}.

For convenience, the \sty{texjavahelp} package provides:
\cmddef{TeXJavaHelpGlsResourceOptions}
This expands to the resource options customized for use with the
\sty{texjavahelp} package. This sets the \opt{entry-type-aliases}
and \opt{assign-fields} options to work with the \ext+{bib} format described in
in \sectionref{sec:bibformat}.

Other resource options included in the definition of
\gls{TeXJavaHelpGlsResourceOptions}
are: \optvalm{selection}{recorded and deps and see}, \optvalm{category}{same as
original entry} and \opt{save-entry-count}. Additionally, this command
starts with:
\cmddef{TeXJavaHelpGlsFieldAdjustments}
which expands to the field adjustment resource options: 
\optval{description-case-change}{firstuc} and \optval{post-description-dot}{check}.
See the \app{bib2gls} user manual for further information on resource options.

\section{The \TeXJavaHelp\ \exttext{bib} Format}
\label{sec:bibformat}

The \ext{bib} format required by \app{bib2gls} is described in the
\app{bib2gls} user manual. The \gls{TeXJavaHelpGlsResourceOptions} command
sets up entry type aliases and field adjustments to assist formatting and work
with the semantic commands described in \sectionref{sec:semanticcmds}.
This means that custom entry types and fields are available that aren't
ordinarily recognised by \app{bib2gls}.

For example, the custom \code{@punctuation} field is used to index
the following symbols in this document:
\glsname{dblqt}, \glsname{percent}, \glsname{tilde},
\glsname{exclam}, and \glsname{bksl}.

\section{Document Division Commands}
\label{sec:docdivisioncmds}

The standard \LaTeX\ sectioning commands such as \gls{chapter}
and \gls{section} should be used. These should all be followed by
\code{\gls{label}\margm{label}}, which allows the section to be
referenced by a context-sensitive help button. The label also
determines the file basename if the section should be split into a
separate file. Any colon prefix will be stripped and all dots will
be replaced with underscores.

For example, this section is started with:
\begin{codebox}
\gls{section}\marg{Document Division Commands}
\gls{label}\marg{sec:docdivisioncmds}
\end{codebox}
So the section label is \code{sec:docdivisioncmds} but the
\qtt{sec:} prefix will be stripped when determining the filename. So
the filename for this section has the basename \filefmt{docdivisioncmds}
(the file extension is determined by \app{texjavahelpmk}['s]
\switch{html-ext} option).

The navigation tree is determined by the table of contents \ext{toc}
file. However, there are some divisions that are not typically
included in the table of contents (such as the table of contents
itself) that need to be in the navigation tree.
Some commands are provided to ensure that nodes are added to the
navigation tree where there is no corresponding entry in the
\ext{toc} file.

\cmddef{WriteTJHDivisionInfo}
This simply writes a line to the \ext{aux} file, which
\app{texjavahelpmk} can pick up. This should be placed immediately
after the section heading. For example:
\begin{codebox}
\gls{chapter}*\marg{Foreword}
\gls{WriteTJHDivisionInfo}\marg{chapter}\marg{}\marg{Foreword}\marg{foreword}\marg{\cmd{thepage}}
\end{codebox}
The \meta{unit} argument should be the sectional unit
(\qtt{chapter}, \qtt{section} etc). The \meta{prefix} argument is
the title prefix, if applicable. This should be empty if there's no
prefix (which is more likely to be the case as the prefix is
typically the chapter or section number). The \meta{target} is a
label that will be used as the basis of the node filename.
The \meta{location} argument will typically be the page number, but
it's mostly ignored.

If \cls{scrbook} or \cls{scrreport} have been loaded, this command
will be used after the table of contents title. It's automatically added with:
\begin{compactcodebox}
\cmd{AfterTOCHead}\oarg{toc}
 \marg{\gls{WriteTJHDivisionInfo}\marg{chapter}\marg{}\marg{\cmd{contentsname}}\marg{toc}\marg{\cmd{thepage}}}
\end{compactcodebox}
If KOMA-Script's \optfmt{toc=listof} option is used to add the list
of figures and list of tables to the table of contents, then this
adjustment won't be necessary for those lists.

\cmddef{HeadlessSection}
This is a shortcut command that creates a hyperlink target and uses
\gls{WriteTJHDivisionInfo} where a new navigation node should start
but there's no associated title. The \meta{title} argument is the
title for the navigation node and does not appear in the document.
The other arguments are as for \gls{WriteTJHDivisionInfo}.

\envdef{prefrontmatterblock}
A shortcut environment that's only available if \gls{chapter} is
defined. This simply does:
\begin{compactcodebox*}
\gls{clearpage}
\gls{HeadlessSection}\margm{target}\marg{chapter}\margm{title}
\gls{pagestyle}\marg{empty}
\meta{body}
\gls{clearpage}
\end{compactcodebox*}

For example:
\begin{codebox}
\cbeg{prefrontmatterblock}\marg{note}\marg{Note}
Some untitled front matter note on a page with no header or footer.
\cend{prefrontmatterblock}
\end{codebox}

\cmddef{WriteTJHLabel}
This is used to establish where a label should actually be for cases
where the label, created with \gls{label}, might float to a different division. The
floating commands \gls{FloatFig}, \gls{FloatSubFigs} and \gls{FloatTable}
all use this command to counteract the possibility that the floats
may end up in a different section. If you directly use floating
environments you may find that the HTML anchor corresponding to the
label is associated with the wrong HTML file. If this happens, you
will need to place \gls{WriteTJHLabel} at the place where the float
(which won't float in the HTML file) will occur in the HTML file.

\begin{important}
\gls{WriteTJHLabel} must be used outside of the floating
environment. The actual label must still be created with
\gls{label}.
\end{important}

For example:
\begin{codebox}
\gls{WriteTJHLabel}\marg{fig:example}
\cbeg{figure}[p]
\gls{includegraphics}\marg{example-image}
\gls{caption}\marg{Example Image}
\gls{label}\marg{fig:example}
\cend{figure}
\end{codebox}

\section{Semantic Commands}
\label{sec:semanticcmds}

Semantic commands are commands that relate to a particular idea or meaning.
For example, \code{\gls{emph}\margm{text}} is a semantic command that indicates
emphasis. This may render \meta{text} in an italic font, but this isn't always the
case. In \gls{html}, this is equivalent to using
\code{\xmltagfmt{em}\meta{text}\xmltagfmt{/em}}.
The style may be changed, but the meaning remains.

As with other commands, semantic commands still have a defined syntax, but
there is no meaning associated with purely syntactic commands, such as
\gls{textit}. The commands \gls{emph} and \gls{textit} have the same syntax, in that
they both take a single argument, but \gls{emph} specifically means that the
argument should be emphasized, whereas \gls{textit} is a font changing
instruction.

The \sty{texjavahelp} package provides a number of semantic commands, not only
because it's good practice but also because it helps the \gls{TeXParserLib} to
produce more accessible \gls{html} with as little inline styling as possible.
The other reason is to allow the use of label prefixes to help disambiguate
closely related labels.  For example, the label \qtt{switch.help} is for the
command line switch \switch{help} whereas \qtt{menu.help} is for a \menu{help}
menu and \qtt{button.help} is for a \widget{help} button.
Buttons and menu items may have mnemonics, such as
\mnemonic{help} for the help button or \mnemonictrail{help.manual}
for the \menu{help.manual} menu item.

\section{Floats}
\label{sec:floats}

Floating environments, such as \env{figure} or \env{table}, can
cause \app{texjavahelpmk}['s] referencing system to fail if they
float to a different section. The associated anchor may end up being
associated with the wrong HTML filename.

The \sty{texjavahelp} package provides some commands to help correct
anchors and provide HTML code that can be more easily adjusted via
CSS styling.

\cmddef{FloatFig}
Creates a \env{figure} environment with the given content centred
and followed by the provided caption and label. The \meta{pos}
argument, if set, will be provided as the \env{figure} option
argument. The \meta{lof} optional argument is for the list of
figures. If present, it will be inserted as an optional argument to
\gls{caption}. The \meta{content} is the figure content, typically
an image included with \gls{includegraphics} or \gls{includeimg}.
The \meta{caption} argument is the caption text and \meta{label}
is the caption label.

The \meta{\LaTeX\ only pre-content} is added immediately
before the \meta{content} (inside the \env{figure}) but only for the
PDF version.
The \meta{\LaTeX\ only pre-content} should not contain anything that would
upset the indexing or cross-references. It's mainly intended to
allow changes to the \sty{graphicx} settings just for the PDF.
It will be ignored by \app{texjavahelpmk}. For example:
\begin{codebox}
\gls{FloatFig}\oarg{p}\marg{fig:sample}
 \oarg{\cmd{setkeys}\marg{Gin}\marg{width=\cmd{textwidth}}}
 \marg{\gls{includegraphics}\marg{example-image}}
 \marg{An Example Image}
\end{codebox}
When compiling the document normally, this essentially does:
\begin{compactcodebox}
\gls{WriteTJHLabel}\marg{fig:sample}
\cbeg{figure}\oarg{p}
 \cmd{centering}
 \cmd{setkeys}\marg{Gin}\marg{width=\cmd{textwidth}}
 \gls{includegraphics}\marg{example-image}
 \gls{caption}\marg{An Example Image}
 \gls{label}\marg{fig:sample}
\cend{figure} 
\end{compactcodebox}
However, \app{texjavahelpmk} will create a \xmltagfmt{div} element
with the \optfmt{id} attribute set to \code{fig:sample}.
(The colon will be stripped if the XML setting is on.)
This should mean that hyperlinks to the figure will scroll to the
top of the figure rather than to the caption (which would cause the
figure content to scroll out of sight).

\cmddef{FloatSideBySideFigs}
Creates two side-by-side figures within a single \env{figure}
environment. Each figure is encapsulated within a \env{minipage}
environment with the width calculated as half the text width less
the side-by-side separator dimension. The content of each
\env{minipage} is centred and followed by the corresponding caption and label.

The \meta{pos} optional argument, if not empty, is the optional
argument to pass to the \env{figure} environment. 
The \meta{\LaTeX\ only pre-both} argument is placed after the start of the
\env{figure} environment and before the first \env{minipage}. 
This argument is ignored by \app{texjavahelpmk} and may be used to
adjust the \sty{graphicx} parameters just for the PDF.

The \meta{caption1} and \meta{caption2} arguments are the text for
the first and second captions. If different text is required for the
list of figures, this can be specified with the optional
\meta{lof1} and \meta{lof2} arguments.
The \meta{label1} and \meta{label2} arguments are the label for the
first and second caption, respectively.

The content for the first figure is specified with the
\meta{content1} argument. If the optional \meta{\LaTeX\ only pre-content1} 
argument is provided, this will be placed immediately before
\meta{content1} but will be ignored by \app{texjavahelpmk}.

The content for the second figure is specified with the
\meta{content2} argument. If the optional \meta{\LaTeX\ only pre-content2} 
argument is provided, this will be placed immediately before
\meta{content2} but will be ignored by \app{texjavahelpmk}.

The \meta{\LaTeX\ pre-figure content\slash HTML between figures}
argument is placed before the start of the \env{figure} environment
for the PDF and between the two figures for the HTML. 
This allows the PDF to be more compact.

\cmddef{floatsidebysidevalign}
The vertical alignment for the two \env{minipage} environments in
\gls{FloatSideBySideFigs} is obtained by expanding
\gls{floatsidebysidevalign}.

\cmddef{FloatSubFigs}
Creates a figure with sub-figures as the content. The starred
version puts the first sub-figure on a line of its own. The
sub-figures will all be placed inside the \env{figure} environment.
The optional \meta{pos} argument, if present, will be passed as the
optional argument to the \env{figure} environment.

The \meta{caption} argument is the caption text for the entire figure.
If alternative text is required for the list of figures, this can be
specified with the optional \meta{lof} argument. The \meta{label}
argument is the label for the caption of the entire figure.

Within the \env{figure} environment the \ctr{subfigure} counter will
be reset to 0 and the content will be centred. The \meta{\LaTeX\
only pre-content} will be inserted before the sub-figures, but only
for the PDF. This argument will be ignored by \app{texjavahelpmk}.
It can be used, for example, to locally change the \sty{graphicx}
parameters just for the PDF.

The \meta{sub-figure list} argument should be a comma-separated list
where each item in the list must be in the form
\begin{compactcodebox}
\margm{label}\margm{content}\oargm{hyperref title}\margm{caption}
\end{compactcodebox}
where \meta{label} is the sub-figure label, 
\meta{content} is the sub-figure content, and
\meta{caption} is the caption text (which may be empty).
If the \meta{hyperref title} optional argument is present, 
\gls{@currentlabelname} will be set to \meta{hyperref title}
otherwise it will be set to \meta{caption}.

For example:
\begin{codebox}
\gls{FloatSubFigs}\marg{fig:samplesubfigs}
\oarg{\cmd{setkeys}\marg{Gin}\marg{width=0.45\cmd{linewidth}}}
\marg{
 \marg{fig:samplesubfigsa}\marg{\gls{includeimg}\marg{imagea}}\marg{},
 \marg{fig:samplesubfigsb}\marg{\gls{includeimg}\marg{imageb}}\marg{}
}
\oarg{An Example Figure With Sub-Figures}
\marg{An Example Figure With Sub-Figures:
\gls{subfigref}\marg{fig:samplesubfigsa} first sub-figure,
\gls{subfigref}\marg{fig:samplesubfigsb} second sub-figure.}
\end{codebox}

Each sub-figure is formatted with:
\cmddef{SubFigureContent}
where the arguments are obtained from the corresponding item in the
\meta{sub-figure list} argument of \gls{FloatSubFigs}.
This uses a \env{tabular} environment to centre the sub-figure image content
above the caption. This command is also responsible for incrementing
the \ctr{subfigure} counter.

A sub-figure can be referenced with \gls{ref} (or \gls{figureref}
etc) but there it will include the parent figure number. If you only
want the actual sub-figure reference (which by default is a
lowercase letter: a, b, etc) then you can use:
\cmddef{subfigref}
This is like \gls{ref} but omits the parent figure number prefix.
The sub-figure reference is formatted with:
\cmddef{subfigurefmt}
This simply does \code{(\cmd{emph}\margm{text})}.

\cmddef{FloatTable}
This command is analogous to \gls{FloatFig} but places its content
inside a \env{table} environment. In this case the content would
typically be a \env{tabular} environment. Again, the purpose of this
command, rather than explicitly using the \env{table} environment,
is to assist \app{texjavahelpmk} in creating cleaner HTML code
(as well as using \gls{WriteTJHLabel} in case the table floats
outside of the current section).

The optional \meta{pos} argument, if present, will be passed as the
optional argument to the \env{table} environment. The \meta{\LaTeX\
only pre-content} is placed at the start of the \env{table}
environment and may be used to make localised adjustments, for
example, to change the value of \gls{tabcolsep}. This argument is
ignored by \app{texjavahelpmk}.
The \meta{caption} argument is the caption text. If the \meta{lot}
argument is present, this will be used for the list of tables
instead of the \meta{caption}. The \meta{label} argument is the
caption label.

If you have a long table that needs to span multiple pages, you can
instead use:
\cmddef{NoFloatLongTable}
You will need to load the \sty{longtable} package if you want to use
this command as it will use the \env{longtable} environment.
The optional \meta{h-align} argument, if present, will be added as
the optional argument to the \env{longtable} environment.
The \meta{label}, \meta{\LaTeX\ only pre-content}, \meta{caption}
and \meta{lot} arguments are as for \gls{FloatTable}.

The \meta{content} argument is different from the corresponding
argument in \gls{FloatTable} as now it shouldn't include the
\env{tabular} environment. Instead, the column specifiers should be
provided in the \meta{col-specs} argument, the header row should be
provided in the \meta{header row} argument and the table body should
be in the \meta{content} argument.

\gls{NoFloatLongTable} will automatically set the \env{longtable}
head and foot. Each foot, except for the last, will have
\continuednextpage. This is encapsulated with:
\cmddef{ContinuedNextPageBox}
This uses \gls{multicolumn} to span all columns, with the \code{r}
alignment specifier and the content is \meta{text} encapsulated
with:
\cmddef{continuedfmt}
This simply does \code{\cmd{emph}\margm{text}}.

Each head, apart from the first, will have \continuedname\ appended
to the caption.  Again, this is encapsulated with
\gls{continuedfmt}.

If you need to stack text vertically within a table cell to make it
more compact for the PDF, you can use:
\cmddef{tablecellstack}
This is simply defined to do \code{\cmd{shortstack}\margm{content}}
by the \sty{texjavahelp} package, but with \app{texjavahelpmk} the
new row \gls{dbbksl} markup will be replaced with a space.

Some convenient commands are provided for cross-referencing.
\cmddef{figureref}
Used to reference figures, this is a shortcut command that uses
\gls{ref} or \starredcs{ref} and is prefixed by:
\cmddef{figurerefprefix}
This simply expands to \gls{figurename} followed by a non-breakable
space.

\cmddef{Figureref}
Used to reference figures, this is a shortcut command that uses
\gls{ref} or \starredcs{ref} and is prefixed by:
\cmddef{Figurerefprefix}
This simply expands to \gls{Figurename} followed by a non-breakable
space.

\section{Image Commands}
\label{sec:imagecmds}

The \sty{graphicx} package is automatically loaded so you can use
\gls{includegraphics} (the definition with the \keyvallist\ optional
argument). The \gls{TeXParserLib} supports this command (but not
all options). The \sty{tikz} package is also automatically loaded
but is not supported by the \gls{TeXParserLib}.

\cmddef{TeXParserLibToImage}
This just does \meta{\LaTeX\ code} when compiling with
\LaTeX\ but the \gls{TeXParserLib} will attempt to convert
the supplied code to an image if \app{texjavahelpmk} is run 
with the default \switch{convert-images} setting.
Recognised options:

\optiondef{TeXParserLibToImage.alt}
Sets the alt text. If not supplied, the \LaTeX\ code will be
used.

\optiondef{TeXParserLibToImage.pre-alt}
If set, inserts \meta{text} at the start of the alt text.

\optiondef{TeXParserLibToImage.post-alt}
If set, inserts \meta{text} at the end of the alt text.

\optiondef{TeXParserLibToImage.name}
Set the image file basename to \meta{basename}. If not
set, a generic basename is used in the form \metafilefmt{img}{n}{}
(for example, \filefmt{img000001.png}). The file extension is
determined by the image type.

\optiondef{TeXParserLibToImage.type}
Identifies the type of image to create, where
\meta{mime-type} should be a supported image MIME type
(\code{image/png}, \code{image/jpeg} or \code{application/pdf}).
In the case of PDF, the image will be embedded in the HTML file
using the \code{object} element. Note that this is not supported
by HTMLDocument. However, a PNG file will be created to use as a
fallback.

\optiondef{TeXParserLibToImage.crop}
A boolean option that determines whether or not to crop the image.

\optiondef{TeXParserLibToImage.div}
Indicates whether or not to encapsulate the image in a
\code{div} element. The value may be boolean (\optfmt{true} or
\optfmt{false}) or the CSS class name.

\cmddef{TeXParserLibToImagePreamble}
If defined, the value of this token list variable will 
be added to the document preamble used to create the image for
\gls{TeXParserLibToImage}.

\cmddef{includeimg}
Behaves like \gls{includegraphics} if \metafilefmt{}{basename}{.tex}
does not exist, otherwise \gls{includeimg} sets the input path to
the same as for \gls{includegraphics} and inputs
\metafilefmt{}{basename}{.tex} (where the file is expected to
include \LaTeX\ image code, such as \sty{tikz} commands).
With \app{texjavahelpmk}, a bitmap will be searched for with the
same basename, otherwise
it the default \switch{convert-images} setting is on,
\app{texjavahelpmk} will convert the image to a bitmap.

\cmddef{includetimg}
As \gls{includeimg} but top-aligns the image.

The options for \gls{includeimg} and \gls{includetimg} are a subset
of those for \gls{includegraphics}: \opt{includeimg.scale},
\opt{includeimg.width}, \opt{includeimg.height},
\opt{includeimg.rotate}, \opt{includeimg.alt},
\opt{includeimg.actualtext}, and \opt{includeimg.artifact}.
Additionally, the \opt{includeimg.frame} boolean option is also
available. If true, this will put a frame around the image.

\chapter{The \apptext{texjavahelpmk} Command Line Application}
\label{sec:texjavahelpmk}

\appdef{texjavahelpmk}

Use the \switch{help} switch 
(or \swch{help}) for help and the
\switch{version} switch
(or \swch{version}) for the version information.

\switchdef{help}
Display help message and exit.

\switchdef{version}
Display version information and exit.

\switchdef{in}
Specifies the input file. The switch may be omitted (that is,
\meta{tex-file} may just be provided) provided that
\switch{output} is also used or the input file is specified before
the output directory.

\switchdef{output}
Specifies the output directory. The switch may be omitted
(that is, \meta{out-dir} may just be provided) provided that
\switch{in} is also used or the output directory is specified after
the input file.

\switchdef{html-ext}
The extension to use for the HTML files.

\switchdef{split}
Specifies the sectioning level to split on. The level should be a
number that corresponds to the section level (0 for chapter, 1 for
section, etc).

Each section should be labelled as the HTML filenames
are constructed from the label. Any colon prefix such as
\code{sec:}\ or \code{ch:}\ will be stripped and any periods will be
removed or replaced by underscores. If a section isn't labelled, the basename
will be \metafilefmt{node}{n}{} instead.

\switchdef{prefix-split}
The HTML filename will be prefixed with the input file basename.
For example, if the input file is \filefmt{foo.tex} and the
introduction section has been assigned the label \code{sec:intro}
then the HTML file for the introduction section will be
\filefmt{foo-intro.html} instead of just \filefmt{intro.html}.

\switchdef{noprefix-split}
Don't prefix the HTML filename with the input file basename.

\switchdef{out-charset}
Sets the character encoding for the output files.

\switchdef{image-dest}
Specifies where to put output images. The given directory should be
relative to the output directory.  If \meta{directory} is an empty
string (the default) then the images will be copied to a location
relative to the output directory that retains the same structure as
for the input. 

For example, if the input file \filefmt{doc.tex}
is in the directory \filefmt{doc/} and the image \filefmt{img1.png} is in
\filefmt{doc/images1/} and the image \filefmt{img2.png} is in
\filefmt{doc/images2/} and, if the output directory is
\filefmt{helpset/}, then with the default setting, \filefmt{img1.png}
will be copied to \filefmt{helpset/images1/} and \filefmt{img2.png}
will be copied to \filefmt{helpset/images2/}. However, if
\switch{image-dest} sets the image destination to \filefmt{images}
then both \filefmt{img1.png} and \filefmt{img2.png} will be copied
to \filefmt{helpset/images/}.  

\switchdef{convert-images}
Use \pdfLaTeX\ or \LuaLaTeX\ to convert \LaTeX\ code (with
\gls{TeXParserLibToImage} or input with \gls{includeimg} etc) to
bitmap images. This is the default. The temporary files used in this
operation will be deleted by default. Use \switchdef{no-rm-tmp-dir}
to retain them.

\switchdef{noconvert-images}
Don't convert \LaTeX\ image files.

\switch{image-preamble}
If an image needs to be automatically generated, the temporary
\LaTeX\ file used to create the image should have the preamble set
to \meta{\LaTeX\ code}. If not set, the preamble will be the same as
the preamble from the input file.

\switch{image-preamble-from-file}
As \switch{image-preamble} but the content is obtained from the file
identified by \meta{filename}.

The following options govern the HTML output.

\switchdef{entities}
Use HTML entities for non-ASCII characters. This will interfere with
the helpset search data.

\switchdef{noentities}
Don't use HTML entities for non-ASCII characters (default).

\switchdef{support-unicode-script}
Use Unicode superscripts or subscripts where available.

\switchdef{nosupport-unicode-script}
Don't use Unicode superscripts or subscripts, just
use \code{sub} or \code{sup} elements (default).

\switchdef{head}
Add \meta{html} code to the HTML header.

\switch{body-preamble}
Insert \meta{html} at the start of the HTML body.

\switch{body-preamble-from-file}
As \switch{body-preamble} but the content is obtained from the file
identified by \meta{filename}.

\switch{body-postamble}
Insert \meta{html} at the end of the HTML body.

\switch{body-preamble-from-file}
As \switch{body-preamble} but the content is obtained from the file
identified by \meta{filename}.

\switchdef{helpset}
Create helpset files. These files are designed for integration with
the \TeX\ Java Help system, which means that there won't be any
HTML5 markup (as it's not supported by Java's \code{HTMLDocument} class).
There also won't be any navigation added as that's dealt with by the
Help window (see \sectionref{sec:helpwindows}).
This setting automatically implements
\switch{nobreadcrumbtrail}, \switch{nominitoc} and
\switch{nomathjax}.

\switchdef{html}
Indicates that the HTML files are not intended for use with the
\TeX\ Java Help system, so HTML5 markup will be used and a
navigation line will be added to the end of each HTML body. This setting
also automatically implements \switch{breadcrumbtrail},
\switch{minitoc} and \switch{mathjax}.

\switchdef{nohelpset}
Deprecated synonym of \switch{html}.

\switchdef{epub}
This is still experimental. This switch indicates that an EPUB file
should be created. Since this format requires XHTML, this setting
will switch on \app{texjavahelpmk}['s] XML mode. All the HTML and
CSS files will be created in the target directory (but not the XML
files that would ordinarily be created with \switch{helpset}) and
also the additional files required by the EPUB format. The EPUB file
will be created using these files (but it will also contain the
\filefmt{mimetype} file which won't be created separately). 

This option automatically implements \switch{nomathjax},
\switch{nominitoc} and \switch{nobreadcrumbtrail}.

\begin{important}
The XML format has restrictions on the characters permitted as
identifiers so \app{texjavahelpmk} will strip or replace problematic
characters in labels but this isn't guaranteed to always work.
\end{important}

\switchdef{mathjax}
Add MathJax support. Note that Java's \code{HTMLDocument} class
doesn't support MathJax, so this setting is off by default.

\switchdef{nomathjax}
Don't add MathJax support. This will just use an HTML
approximation which is quite limited.

\switch{breadcrumbtrail}
Add breadcrumb trail to the start of HTML body.

\switch{nobreadcrumbtrail}
Don't add breadcrumb trail to the start of HTML body.

\switchdef{minitoc}
Add minitoc to HTML body. This isn't required for helpset files as
the Help window provides a navigation panel.

\switchdef{nominitoc}
Don't add minitoc to HTML body.

\switchdef{minitoc-div-class}
If \meta{class-name} not empty, encapsulate minitoc in a
div with the given class name. This option is ignored with
\switch{nominitoc}.

\switchdef{minitoc-div-id}
If \meta{id-name} not empty, encapsulate minitoc in a
div with the id class name. This option is ignored with
\switch{nominitoc}.

If neither \switch{minitoc-div-class} nor \switch{minitoc-div-id} is
set to a non-empty value, then the encapsulating div won't be added.

\switch{minitoc-preamble}
Insert \meta{html} at the start of the minitoc (inside the
encapsulating div if \switch{minitoc-div-class} or \switch{minitoc-div-id} have been
set).

\switch{minitoc-preamble-from-file}
As \switch{minitoc-preamble} but the content is obtained from the file
identified by \meta{filename}.

\switch{minitoc-postamble}
Insert \meta{html} at the end of the minitoc (inside the
encapsulating div if \switch{minitoc-div-class} or \switch{minitoc-div-id} have been
set).

\switch{minitoc-postamble-from-file}
As \switch{minitoc-postamble} but the content is obtained from the file
identified by \meta{filename}.

The following options are provided for debugging purposes.

\switchdef{no-rm-tmp-dir}
Don't remove temporary files on exit.

\switchdef{rm-tmp-dir}
Remove temporary files on exit (default).

\switchdef{log}
Set the \TeX\ Parser log file to \meta{filename}.

\switchdef{nolog}
Don't log messages.

\switchdef{debug}
Switch on debug mode. The \meta{level} (if present) should be an
integer greater than 0. (Setting the level to 0 is equivalent to

\switchdef{debug-mode}

Enable \TeX\ Parser debugging. Be aware that this can be extremely
verbose.

\switchdef{nodebug}
Switch off debugging mode (default).

\switchdef{no-debug}
Synonym for \switch{nodebug}.

\chapter{The \TeX\ Java Help Library (\filetext{texjavahelplib.jar})}
\label{sec:texjavahelplib}

The \inlineglsdef{file.texjavahelplib.jar} file contains the 
\TeX\ Java Help Library classes that may be used to load the
files created by \app{texjavahelpmk} and display the information in
a frame with navigation buttons. The library also deals with loading
localisation files, so \app{tjhxml2bib}, \app{texjavahelpmk} and the
application requiring the \TeX\ Java Help Library's \gls{gui}
elements need to have \file{texjavahelplib.jar} on the class path.

The library also comes with images that are used as icons in the
help frame buttons and \glspl{menu}. Text and icons for common buttons 
\widget+{okay}, \widget+{apply}, \widget+{close} and
\widget+{cancel} are provided, but may be overridden by
providing alternative values in the application language file
or in the application's icon directory.

The keyword search will only work if \app{texjavahelpmk} has created
the search \gls{xml} file, which catalogues the search terms with a
reference to the file and block or paragraph where the terms were
found. Similarly, the index is created during the document build
process, which depends on \sty{glossaries-extra} and \app{bib2gls}
to create the required files that are parsed by \app{texjavahelpmk}.

The file \filefmt{helpinterface.tex} provides a description of the
help system and is designed to be input into the application's
documentation source file (as in \sectionref{sec:helpwindows}).

The main help frame can't be accessed when a modal dialog is open.
There is therefore an interface to allow just the pages applicable
to a dialog to be viewed. This means that a help button in the
dialog box can open a minimal view. If the main help frame happens
to be open at the same time, it will be closed when the help dialog
is opened to avoid confusion. The help dialog's page will be added
to the main help frame's history so that the user can go back to it
after the dialog has been closed by simply reopening the main help
frame.
